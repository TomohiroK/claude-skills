## 学習事項

### 失敗パターン

#### F1: サーバーレス環境でのネイティブ依存ライブラリ

- **事象**: isomorphic-dompurify -> jsdom -> canvas（ネイティブバイナリ）の依存チェーンが Vercel サーバーレスで動作しない
- **根本原因**: `npm install` + `next build` がローカルで成功しても、Vercel のサーバーレスランタイムはネイティブバイナリを含むパッケージを正しくバンドルできない場合がある
- **教訓**: Server Component で使うライブラリを追加する際は、以下を確認する:
  1. ネイティブバイナリ依存（node-gyp, canvas, sharp 等）がないか
  2. Node.js API のうちサーバーレスで制限されるもの（fs の一部、child_process 等）を使っていないか
  3. Vercel プレビューデプロイで動作確認してから本番デプロイする
- **判断**: ルール反映を推奨（下記参照）

#### F2: デプロイ後のヘルスチェック欠如

- **事象**: 本番で 500 エラーが発生していたが、ユーザー報告まで気づかなかった
- **教訓**: デプロイ後に主要ページの HTTP ステータスを自動確認する仕組みが必要
- **判断**: development-workflow.md への追記を推奨

### 成功パターン

#### S1: データ駆動アーキテクチャ

- posts.json にブログデータを集約し、リポジトリパターンで CRUD を提供する設計は、週次自動公開タスクとの連携に適していた
- 3x3-portal の既存パターン（event.repository.ts, team.repository.ts）と一貫性がある

#### S2: コードレビューによる XSS 防止

- dangerouslySetInnerHTML のサニタイズ漏れをコードレビューで検出できた
- Phase 7 を省略していれば、XSS 脆弱性が本番に到達していた可能性がある

## ルール反映提案

### 提案1: Vercel サーバーレス互換性チェック（nextjs.md への追記）

```markdown
## サーバーレス環境でのライブラリ互換性

### Server Component に新規ライブラリを追加する際の確認事項
- ネイティブバイナリ依存（node-gyp, canvas, sharp 等）がないか `npm ls` で確認する
- jsdom 等のブラウザエミュレーション系ライブラリはサーバーレスで動作しない場合がある
- ローカルの `next build` 成功は Vercel サーバーレスでの動作を保証しない
- 新規ライブラリ追加後は Vercel プレビューデプロイで動作確認してから本番デプロイする

### 発生事例（2026-03-28）
isomorphic-dompurify が jsdom に依存しており、Vercel サーバーレスで 500 エラー。ローカルビルドは成功していた。
ブログデータが自前 JSON（ユーザー入力経路なし）であったため、サーバーサイド sanitization 自体が不要と判断し削除で解決。
```

### 提案2: デプロイ後ヘルスチェック（development-workflow.md への追記）

```markdown
## デプロイ後のヘルスチェック（必須）
- 本番デプロイ後、変更対象ページの HTTP 200 OK を確認する
- 確認方法: `curl -s -o /dev/null -w "%{http_code}" https://example.com/path`
- 500/502/503 が返った場合は即座にロールバックする
- 「ビルド成功 = 本番正常」と判断しない
```
