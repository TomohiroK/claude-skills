## Problem（発生した問題）

### P1. DesktopNav Client Component追加によるランタイムエラー [担当: CTO] -- 最重大

**何が起きたか**:
- SiteHeader（Server Component）内に新規のDesktopNav（Client Component）を追加
- `npm run dev --prefix` 経由の起動環境でwebpackのモジュール解決が壊れた
- `TypeError: Cannot read properties of undefined (reading 'call')` が全ページで発生
- 最終的にDesktopNavを廃止し、Server Componentの静的リンクに戻して解決

**根本原因（3層）**:
1. **設計判断の誤り**: ナビのアクティブ状態表示のためだけに新規Client Componentを追加するのは過剰だった。Server Componentの静的リンクで十分な要件
2. **--prefix環境の理解不足**: `npm run dev --prefix` がwebpackのcwdに影響し、Client Componentのモジュール解決を壊す可能性を認識していなかった
3. **原因切り分けの手順ミス**: エラー発生時に最初にすべき「自分の変更を一旦戻して再現確認」をせず、キャッシュクリアや再起動を6回以上繰り返した

**影響**:
- デバッグに大幅な時間を浪費
- 最終的に当初の実装を全て廃止する手戻り

### P2. エラーの外部要因への誤帰属 [担当: CTO]

**何が起きたか**:
- 自分のコード変更が原因のランタイムエラーを「プレビューツールのキャッシュ問題」と決めつけた
- `.next`キャッシュ削除、サーバー再起動を何度も繰り返したが解決せず
- `git stash` で元コードに戻して初めて「自分の変更が原因」と判明

**根本原因**:
- 「エラーの原因は自分のコードか、環境か」の切り分けを最初に行わなかった
- 確証バイアス: 「自分のコードは正しいはず」という思い込みにより、環境要因を先に疑った

### P3. Next.js Server/Client Component境界の軽視 [担当: CTO]

**何が起きたか**:
- Server Component内にClient Componentを埋め込む際の制約（'use client'ディレクティブ、propsのシリアライズ制約、webpack bundleへの影響）を十分に検討せずに実装した

**根本原因**:
- Next.js App Routerにおける Server Component → Client Componentの境界は、単純な「import + 'use client'」では済まないケースがある
- 特に `--prefix` のような非標準的な実行環境では、bundlerの挙動が変わりうる

## 過去の学びとの関連

前回（2026-03-24）の振り返りで以下を学んだ:
