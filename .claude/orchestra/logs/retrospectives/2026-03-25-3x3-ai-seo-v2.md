## Problem（発生した問題）

### P1. 英語版の必要性を事前に想定できなかった [担当: CMO/CTO]

**何が起きたか**:
- セッション1-2で日本語版のみ作成。セッション3でユーザーから「LLMは英語で学習・推論するため、英語版も必要」と指摘された

**根本原因**:
- llms.txt の目的は「AIが情報を正確に取得すること」であり、ターゲットは人間ではなくLLM
- LLMの学習データの大部分が英語であること、推論時も英語の方が精度が高いことを考慮しなかった
- 日本語サイトだから日本語で書く、という「人間向けSEO」の思考に引きずられた

**影響**:
- 中（セッション1追加分の工数。ただし日本語版をテンプレートにできたため効率的だった）

**教訓**:
- AI向けコンテンツは「AIがどう処理するか」を基準に言語・構造を設計する。人間向けコンテンツの延長で考えない

### P2. Next.js metadata API の限界に関する事前調査不足 [担当: CTO]

**何が起きたか**:
- `metadata.alternates.types` が任意のMIME typeに対応しないことを、実装・デプロイ後に発見

**根本原因**:
- フレームワークAPIの対応範囲を公式ドキュメントで事前確認せず、「おそらく動く」で進めた

**影響**:
- 小（デプロイ1回分の手戻り。対処は迅速だった）

**対処済み**:
- nextjs.md ルールに「metadata APIの対応範囲」セクションを追記済み（前回振り返りで対応）

### P3. ローカルdevサーバーの制限事項の文書化不足 [担当: CTO]

**何が起きたか**:
- `--prefix` 環境でlayout.tsx変更後にHMRが機能せず黒画面。devサーバー再起動が必要
- 本番では正常動作するため混乱の原因になりうる

**根本原因**:
- `--prefix` 環境の制限事項が一部しか文書化されていない（Client Component関連のみ）
- layout.tsx等のルートレベル変更でもHMRが壊れるケースがある

**影響**:
- 小（再起動で解決。ただし知らないと混乱する）
