# 学習ログ - 2026-03-24

**セッション**: fixedassets ブランディング作業（Ledgea命名・ロゴ・ブランドガイドライン策定・適用）

---

## 成功パターン

### CxOボードの並行起動
- **何がうまくいったか**: CMO・CVO・CSOを同時並行で起動し、各専門領域の分析を同時に得た後、CEOが統合決裁する流れが効率的だった
- **なぜうまくいったか**: 各CxOが独立して専門分析を行えるため、逐次実行より大幅に時間を短縮できた
- **再利用条件**: 意思決定に複数の専門観点（マーケティング・ビジョン・戦略等）が必要な場合に有効

### 多角的命名分析
- **何がうまくいったか**: 「Ledgea」という商品名を言語学的（ASEAN各言語での発音・意味）・市場的（競合との差別化）・法的（商標）の三軸で分析した
- **なぜうまくいったか**: 単なる語感ではなく根拠のある分析により、ユーザーの納得感が高まった
- **再利用条件**: グローバル展開を前提とした商品命名に応用可能

---

## 失敗パターンと根本原因

### パターン1: コードベースの二重読み込み
- **何がうまくいかなかったか**: 同一プロジェクトのコードを探索フェーズと実装フェーズの2回読み込み、トークンを浪費した
- **根本原因**: 探索結果をCLAUDE.mdに記録しなかったため、実装時に再確認が必要になった
- **改善策**: コードベース探索後、即座に重要な構造・設定・依存関係をCLAUDE.mdに箇条書きで記録する

### パターン2: ビルドツール設定の把握不足
- **何がうまくいかなかったか**: Trunkの`copy-file`ディレクティブを把握せずにfaviconを配置し、ビルドに反映されなかった
- **根本原因**: Rustフロントエンドプロジェクト固有のビルドパイプライン（Trunk.toml）を事前に確認しなかった
- **改善策**: Rustプロジェクトでの作業開始前に必ずTrunk.toml・Cargo.toml・launch.json等を確認する

### パターン3: 過剰な確認質問
- **何がうまくいかなかったか**: 作業の節目ごとに「確認しますか？」「進みますか？」と繰り返し質問し、ユーザーの作業リズムを損ねた
- **根本原因**: 自律判断への躊躇いがあり、ユーザーの「進めましょう」という意図を都度確認しようとした
- **改善策**: 明示的な委任（「進めて」「やってください」等）があった場合は完了まで確認なしで進める

### パターン4: 対象リポジトリの誤認
- **何がうまくいかなかったか**: fixedassets-bnという類似名ディレクトリを作業対象と混同しかけた
- **根本原因**: 複数の類似プロジェクトが存在する環境で、対象範囲を最初に確認しなかった
- **改善策**: 類似名ディレクトリが存在する場合、作業開始時にユーザーへ対象範囲を確認する

### パターン5: 議事録の担当者記録漏れ
- **何がうまくいかなかったか**: 最初の議事録の決定事項テーブルに担当CxOを記載せず、ユーザーに指摘された
- **根本原因**: テンプレートに「担当者」列が含まれていたが、記入を怠った
- **改善策**: 決定事項テーブルには「決定内容・担当CxO・期限」の3列を必須とする

# 学習ログ #2 - 2026-03-24

**セッション**: ワンデンタル歯科衛生士紹介キャンペーンLP制作

---

## 成功パターン

### 役務対価構成による景表法対策
- **何がうまくいったか**: 紹介報酬を「景品」ではなく「役務の対価」として構成し、利用規約で業務委託契約として明確化した
- **なぜうまくいったか**: CLOの参加により、景品表示法の規制構造（景品類の定義と適用除外）を正確に把握し、法的に安全な設計ができた
- **再利用条件**: 紹介報酬・アフィリエイト報酬など金銭が絡むキャンペーン全般に応用可能

### 登録番号ベースの紹介メカニズム
- **何がうまくいったか**: 既存の歯科衛生士登録番号を紹介コードとして活用することで、追加システム開発なしに紹介追跡が実現できた
- **なぜうまくいったか**: 既存システムの仕組みを理解し、新規ID発行ではなく既存IDの流用を選択した
- **再利用条件**: 会員制サービスで既にユニークIDを持つ場合に有効

### 画像圧縮の大幅サイズ削減
- **何がうまくいったか**: 7.7MB/8.2MBの画像を161KB/224KBに圧縮（97-98%削減）
- **なぜうまくいったか**: 適切な圧縮ツールと品質設定を使用した
- **再利用条件**: LP制作時、画像素材は必ず圧縮してから配置する

---

## 失敗パターンと根本原因

### パターン6: 要件確認なしの作業着手（最重大）
- **何がうまくいかなかったか**: 依頼者に1つも質問せずにLP制作を開始し、紹介メカニズム・報酬形式を誤った前提で制作した
- **根本原因**: 「自律的に進める」を「質問せずに進める」と誤解。要件確認と作業の自律性は独立した概念
- **改善策**: 作業着手前に必ず要件チェックリストを実行する。ビジネス要件・技術仕様・デザイン要件の3カテゴリは最低限確認
- **影響度**: 大（制作物の根幹部分の手戻り）

### パターン7: 法務の遅延参加
- **何がうまくいかなかったか**: 商用キャンペーンなのにCLOが初期から参加しておらず、法的リスクの検討が遅れた
- **根本原因**: オーケストレーターの召集パターンに「金銭が絡む施策 → CLO必須」のルールがなかった
- **改善策**: キャンペーン/プロモーションの召集パターンを追加し、CLOを初期メンバーに含める
- **影響度**: 大（構造的な設計変更が必要になった）

### パターン8: セキュリティ観点の欠落
- **何がうまくいかなかったか**: フィッシング対策の注意喚起を最初から入れなかった
- **根本原因**: LP制作をマーケティング領域のみと捉え、セキュリティ観点が欠落
- **改善策**: キャンペーンLP制作時はCTO/セキュリティ担当を参加させる
- **影響度**: 中（追加は容易だが、気づかないまま公開されるリスクがあった）

### パターン9: キャンペーンライフサイクルの未設計
- **何がうまくいかなかったか**: 終了ページ（ended.html）を最初から用意せず、依頼者に指摘されて作成
- **根本原因**: キャンペーンの「開始→運用→終了」というライフサイクル全体を最初に設計しなかった
- **改善策**: キャンペーン系LP制作では「開始ページ+終了ページ+利用規約」を最低限の3点セットとする
- **影響度**: 中（追加作業が発生）

### パターン10: 実在しないサービスの記載
- **何がうまくいかなかったか**: 「専任サポート」という存在しないサービスをLPに記載した
- **根本原因**: 競合LPの訴求要素をそのまま流用し、実際のサービス内容を確認しなかった
- **改善策**: LP上の全訴求ポイント・サービス記載は依頼者に事実確認を取る。推測でサービス内容を記載しない
- **影響度**: 大（景表法上の不当表示リスク）

### パターン11: 成立条件の誤認
- **何がうまくいかなかったか**: 「登録+応募」で成立と記載。正しくは「登録のみ」で成立
- **根本原因**: パターン6（要件確認不足）の派生
- **改善策**: キャンペーン成立条件は依頼者に必ず確認する
- **影響度**: 大（ユーザーの期待と実際の条件の乖離）

## プロセステンプレート

### キャンペーンLP制作の標準フロー
```
0. 要件確認（ビジネス要件・技術仕様・デザイン要件を依頼者に質問）
1. CLO参加: 法的リスク事前チェック（景表法・職安法・個人情報保護法等）
2. CMO設計: キャンペーン構造設計（条件・報酬・期間・限定要素）
3. CVO設計: ビジュアル構成（画像配置・写真素材・動きの演出）
4. CTO確認: セキュリティ観点（フィッシング対策・偽サイト注意喚起）
5. 制作: LP本体 + 終了ページ + 利用規約（3点セット）
6. ファクトチェック: 全訴求ポイントの事実確認
7. 画像圧縮・最適化
8. SEO設定（title/description/OGP/構造化データ/canonical）
9. 法務最終レビュー
```

### キャンペーン要件確認チェックリスト
```
ビジネス要件:
- [ ] キャンペーン期間（開始日・終了日）
- [ ] 報酬内容・金額・支払方法
- [ ] 成立条件（何をもって成立とするか）
- [ ] 対象者（誰が参加できるか）
- [ ] 上限・先着の定義と判定基準
- [ ] 不正行為の定義と対処方針
- [ ] 既存サービスの正確な内容

技術仕様:
- [ ] 紹介の追跡方法（リンク・コード・番号等）
- [ ] 既存システムとの連携方式
- [ ] 設置URL・ドメイン
- [ ] 申込・問い合わせの導線

デザイン要件:
- [ ] ブランドカラー・ロゴ・フォント
- [ ] 参考サイト・トーン
- [ ] 使用可能な画像素材
```
