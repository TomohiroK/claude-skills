## 成功パターン

### CxOボードの並行起動
- **何がうまくいったか**: CMO・CVO・CSOを同時並行で起動し、各専門領域の分析を同時に得た後、CEOが統合決裁する流れが効率的だった
- **なぜうまくいったか**: 各CxOが独立して専門分析を行えるため、逐次実行より大幅に時間を短縮できた
- **再利用条件**: 意思決定に複数の専門観点（マーケティング・ビジョン・戦略等）が必要な場合に有効

### 多角的命名分析
- **何がうまくいったか**: 「Ledgea」という商品名を言語学的（ASEAN各言語での発音・意味）・市場的（競合との差別化）・法的（商標）の三軸で分析した
- **なぜうまくいったか**: 単なる語感ではなく根拠のある分析により、ユーザーの納得感が高まった
- **再利用条件**: グローバル展開を前提とした商品命名に応用可能

## 失敗パターンと根本原因

### パターン1: コードベースの二重読み込み
- **何がうまくいかなかったか**: 同一プロジェクトのコードを探索フェーズと実装フェーズの2回読み込み、トークンを浪費した
- **根本原因**: 探索結果をCLAUDE.mdに記録しなかったため、実装時に再確認が必要になった
- **改善策**: コードベース探索後、即座に重要な構造・設定・依存関係をCLAUDE.mdに箇条書きで記録する

### パターン2: ビルドツール設定の把握不足
- **何がうまくいかなかったか**: Trunkの`copy-file`ディレクティブを把握せずにfaviconを配置し、ビルドに反映されなかった
- **根本原因**: Rustフロントエンドプロジェクト固有のビルドパイプライン（Trunk.toml）を事前に確認しなかった
- **改善策**: Rustプロジェクトでの作業開始前に必ずTrunk.toml・Cargo.toml・launch.json等を確認する

### パターン3: 過剰な確認質問
- **何がうまくいかなかったか**: 作業の節目ごとに「確認しますか？」「進みますか？」と繰り返し質問し、ユーザーの作業リズムを損ねた
- **根本原因**: 自律判断への躊躇いがあり、ユーザーの「進めましょう」という意図を都度確認しようとした
- **改善策**: 明示的な委任（「進めて」「やってください」等）があった場合は完了まで確認なしで進める

### パターン4: 対象リポジトリの誤認
- **何がうまくいかなかったか**: fixedassets-bnという類似名ディレクトリを作業対象と混同しかけた
- **根本原因**: 複数の類似プロジェクトが存在する環境で、対象範囲を最初に確認しなかった
- **改善策**: 類似名ディレクトリが存在する場合、作業開始時にユーザーへ対象範囲を確認する

### パターン5: 議事録の担当者記録漏れ
- **何がうまくいかなかったか**: 最初の議事録の決定事項テーブルに担当CxOを記載せず、ユーザーに指摘された
- **根本原因**: テンプレートに「担当者」列が含まれていたが、記入を怠った
- **改善策**: 決定事項テーブルには「決定内容・担当CxO・期限」の3列を必須とする

## プロセステンプレート

### ブランディング作業の標準フロー
```
1. ネーミング（CxOボード並行分析 → CEO決裁）
2. ロゴ設計（バリエーション6点以上 + プレビューHTML）
3. ブランドガイドライン策定（色彩・タイポグラフィ・ASEAN文化配慮）
4. プロジェクトへの適用（index.html / lp.html / locales / ソース）
5. ビルド確認（Trunk.toml等のビルド設定を含む）
6. 検証（ブラウザプレビュー・favicon表示確認）
```

### Rustフロントエンドプロジェクトの事前確認チェックリスト
```
- [ ] Trunk.toml の [[bin]] / [[asset]] / copy-file ディレクティブ
- [ ] Cargo.toml の dependencies と features
- [ ] public/ ディレクトリの構成
- [ ] dist/ の出力パス設定
- [ ] launch.json / Makefile 等のビルドスクリプト
```

## 外部要因からの学び

- ASEANマーケットでのSaaS命名では、英語の意味だけでなくインドネシア語・マレー語・タイ語等での語感・意味も検証が必要
- 「Ledgea」はASEAN各言語で中立的な発音であり、会計（Ledger）の連想も持つ良い命名事例となった
