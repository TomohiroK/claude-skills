## 成功パターン

### S1. innerHTML を排除し DOM API で動的コンテンツを描画する
- **文脈**: articles.json のデータを index.html の Insights セクションに描画する処理
- **パターン**: `textContent` + `createElement` で全要素を生成。slug は `/[^a-z0-9-]/g` でサニタイズ
- **効果**: XSS とパストラバーサルを構造的に防止（レビューで発見し修正）
- **適用範囲**: 外部データ（JSON、API レスポンス）を DOM に描画する全ケース

### S2. モバイル分析は visibilitychange + sendBeacon で実装する
- **文脈**: 記事の滞在時間をGA4で計測する処理
- **パターン**: `beforeunload` → `visibilitychange` イベントに変更。`navigator.sendBeacon` で確実に送信
- **理由**: beforeunload はモバイルブラウザ（特にiOS Safari）で発火しないケースがある
- **適用範囲**: ページ離脱時のアナリティクス送信全般

### S3. 新UI要素は既存デザインパターンのカタログから選ぶ
- **文脈**: $50プライシングバナーがユーザーに「AIっぽい」と指摘された
- **パターン**: サイト内の既存カードパターン（service-tile, process-step）を分析し、同じスタイルで実装
- **効果**: サイト全体のデザイン一貫性を維持。ユーザーから承認
- **適用範囲**: 既存サイトに新セクション/要素を追加する全ケース

### S4. フィードバックループを初期設計に組み込む
- **文脈**: 記事の自動生成システム
- **パターン**: GA4メトリクス → metrics.json → 次回生成時のプロンプト入力。メトリクス収集cronは生成cronの1時間前
- **効果**: 「作って終わり」ではなく、データ駆動の改善サイクルが回る
- **適用範囲**: コンテンツ生成の自動化全般

### S5. HTML → スクリーンショット → 圧縮 による画像生成ワークフロー
- **文脈**: OGP画像、サービスセクション画像の作成
- **パターン**: サイトのデザイントークン（色、フォント、間隔）を使ったHTML → Playwright/Previewでスクリーンショット → sips/圧縮
- **効果**: デザインツール不要でブランド一貫性のある画像を生成。全画像300KB以内
- **適用範囲**: OGP画像、セクションビジュアル、プレゼン素材

## 失敗パターン

### F1. Worksセクションに掲載中のプロダクトがダウンしていた
- **文脈**: Ledgea のスクリーンショット取得時に ledgea.tmkproduct.com が名前解決不能
- **原因**: デプロイ状態の定期確認がない
- **影響**: ローカルLPで代替撮影（本番スクリーンショットの方が望ましい）
- **対策**: Worksに掲載するプロダクトは定期的にアクセス可能性を確認する

### F2. cronタスクの初回実行テストが未完了のままクローズ
- **文脈**: 週次記事生成・メトリクス収集のスケジュールタスク
- **原因**: タスク作成 = 完了 と判断してしまった
- **影響**: 月曜の初回実行でツール権限未承認によるエラーの可能性
- **対策**: cronタスク作成後は初回手動実行テストを完了条件に含める
