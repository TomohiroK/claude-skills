## 失敗パターン

### F1. ブラウザリソースを使い捨てにしてはいけない（AudioContext枯渇）
- **文脈**: 音声再生のたびに `new AudioContext({ sampleRate: 24000 })` を作成し、再生後に `close()` していた
- **症状**: ブラウザのAudioContext上限（6〜8個）に達し、録音用AudioContextがsuspendされてマイクが死亡。2回目以降の会話で声が拾えない
- **根本原因**: ブラウザリソースには上限があるという基本知識の欠如。サーバーサイドの感覚で「作って使って捨てる」を繰り返した
- **正しいパターン**: `let playCtx = null;` で変数を保持し、`if (!playCtx || playCtx.state === "closed") playCtx = new AudioContext()` で再利用
- **適用範囲**: AudioContext, WebSocket, MediaStream, Web Worker, Service Worker 等ブラウザのランタイムリソース全般

### F2. 1回テストでOK判定は素人（マルチラウンドテストの欠如）
- **文脈**: 音声会話機能のテストで、1回目の会話が成功した時点で「動作確認完了」とした
- **症状**: 2回目の会話でマイクが死亡（F1のバグ）。ユーザーに発見された
- **根本原因**: テストの基本「1回目、2回目、n回目」を実践しなかった。状態をリセットして再実行するフローのテストが皆無
- **ユーザーの言葉**: 「テストの基本として１回目、２回目、n回目ってするのが普通。１回動いてからOKは素人すぎる」
- **正しいパターン**: 状態リセットを伴う機能は最低3ラウンド（初回・2回目・3回目以降）テストする。特にリソース管理（接続/切断/再接続）は累積バグが出やすい
- **適用範囲**: WebSocket接続/切断、ログイン/ログアウト、ページ遷移/戻り、モーダル開閉 等、状態リセットを伴う全機能

### F3. リソースクリーンアップは末端から根元へ
- **文脈**: `goBack()` で `audioCtx.close()` を `stopRecording()` より先に呼んでいた
- **症状**: MediaStreamトラックが停止する前にAudioContextを閉じると、トラックが孤児になりリソースリーク
- **根本原因**: クリーンアップの依存関係を考えずに書いた
- **正しい順序**: タイマーキャンセル → 再生キュークリア → stopRecording()（MediaStream停止） → audioCtx.close() → playCtx.close() → ws.close() → UI reset
- **原則**: 「末端（タイマー、子プロセス）→ 中間（ストリーム、コンテキスト） → 根元（接続、UI）」の順で閉じる
- **適用範囲**: WebSocket + Audio/Video + Timer を組み合わせるリアルタイムアプリ全般

### F4. CTOがテンパると品質が崩壊する
- **文脈**: AudioContext枯渇バグ発覚後、焦って場当たり的に修正を重ねた
- **症状**: 修正のたびに新しい問題が発生（順序ミス、タイマー未クリア、transcript未クリア）
- **根本原因**: パニック状態で「とりあえず動かす」モードに入り、設計を無視した
- **正しいパターン**: バグ連鎖が発生したら「止まる→全体のリソース管理を設計し直す→テスト計画を立てる→実装」の手順を踏む
- **適用範囲**: 複数バグが連鎖して発生した全ケース

### F5. QA品質ゼロからの復帰
- **文脈**: 最初のQAは「1回OK→合格」という最低品質だった
- **復帰プロセス**: ユーザー指摘 → 3ラウンドモックテスト設計 → リソース状態（ws/audioCtx/playCtx/isRecording/transcript）の全項目検証 → 合格
- **教訓**: 復帰できたこと自体は良い。しかし最初からこの品質で出すべきだった。「復帰できた」を成功体験にしてはいけない
- **適用範囲**: QA全般

## 成功パターン

### S1. ws.onclose を単一リセットポイントにする設計
- **文脈**: 戻るボタン実装時、goBack() と ws.onclose の責務を分離
- **パターン**: goBack() はリソース閉鎖のみ、ws.onclose がUI状態の全リセットを担う。正常終了もエラー終了も同じコードパスを通る
- **効果**: リセット漏れが構造的に防がれる（1箇所を直せば全パターンに反映）
- **適用範囲**: WebSocket接続を持つ全アプリ

### S2. selectedLang / currentLang の分離パターン
- **文脈**: 言語選択UI（選択画面）と会話中の言語（ロック済み）の状態管理
- **パターン**: `selectedLang` は選択画面で自由に変更可能、`currentLang` は `start()` 時にロックされ会話中は不変
- **効果**: 会話中に言語ボタンを押しても会話が壊れない
- **適用範囲**: 「設定画面の値」と「実行中のセッションで使う値」を分離する全ケース
