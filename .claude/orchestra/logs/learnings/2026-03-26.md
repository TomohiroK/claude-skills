## 失敗パターンと根本原因

### パターン22: JSX属性のキャメルケース変換漏れ
- **何がうまくいかなかったか**: `hreflang` をそのまま書いたが、JSXでは `hrefLang` が正しい。ビルドで型エラー発生
- **根本原因**: HTML属性をJSXにコピーする際、キャメルケース変換のチェックを省略した
- **改善策**: JSXでHTML属性を書く際は `class→className`, `hreflang→hrefLang`, `tabindex→tabIndex` 等の変換を意識する
- **影響度**: 小（ビルドエラーで即検出、修正1行）
- **キーフレーズ**: 「JSXはHTMLではない。属性名はキャメルケースに変換する」

### パターン23: Next.js 動的ルートによる静的ファイルの遮蔽
- **何がうまくいかなかったか**: `public/llms.txt` を配置したが、`[jobType]` 動的ルートが先にマッチし404を返した
- **根本原因**: Next.js App Router では動的ルート `[slug]` が `/public` の静的ファイルより優先される場合がある。`next.config.js` の `rewrites({ beforeFiles })` で回避が必要
- **改善策**: 動的ルートがあるプロジェクトで `public/` に新規ファイルを追加する際は、ルート競合を確認する。競合する場合は `rewrites` で明示的にルーティングする
- **影響度**: 中（ステージングデプロイ後にユーザーが検出・対応）
- **キーフレーズ**: 「動的ルートは静的ファイルを食う。[slug] がある場所に public/ ファイルを追加するなら rewrites で保護する」

### パターン24: ローカル開発環境なしでの変更
- **何がうまくいかなかったか**: `.env` 未設定でdevサーバーが起動せず、ローカル検証をスキップしたため P22, P23 がデプロイ後に発覚
- **根本原因**: 外部プロジェクトのローカル環境セットアップを事前確認しなかった
- **改善策**: `.env` がなくても `npm run build` でTypeScript型チェックは可能。最低限ビルド検証を試みる
- **影響度**: 中（デプロイ失敗→修正→再デプロイのサイクル）
- **キーフレーズ**: 「ローカルで検証できないなら、せめてビルドを通す」

## プロセステンプレート

### 動的ルート + 静的ファイル競合の確認手順
```
1. プロジェクトに [slug] / [...catchAll] 動的ルートがあるか確認
2. ある場合、追加するファイルのパスが動的ルートにマッチしないか確認
   - /llms.txt → app/[jobType] にマッチする可能性あり
   - /sitemap-static.xml → 拡張子が異なるためマッチしない
3. マッチする場合、next.config.js に rewrites を追加:
   rewrites() {
     return {
       beforeFiles: [
         { source: "/llms.txt", destination: "/llms.txt" },
       ],
     }
   }
```
