## 失敗パターンと根本原因

### パターン22: JSX属性のキャメルケース変換漏れ
- **何がうまくいかなかったか**: `hreflang` をそのまま書いたが、JSXでは `hrefLang` が正しい。ビルドで型エラー発生
- **根本原因**: HTML属性をJSXにコピーする際、キャメルケース変換のチェックを省略した
- **改善策**: JSXでHTML属性を書く際は `class→className`, `hreflang→hrefLang`, `tabindex→tabIndex` 等の変換を意識する
- **影響度**: 小（ビルドエラーで即検出、修正1行）
- **キーフレーズ**: 「JSXはHTMLではない。属性名はキャメルケースに変換する」

### パターン23: Next.js 動的ルートによる静的ファイルの遮蔽
- **何がうまくいかなかったか**: `public/llms.txt` を配置したが、`[jobType]` 動的ルートが先にマッチし404を返した
- **根本原因**: Next.js App Router では動的ルート `[slug]` が `/public` の静的ファイルより優先される場合がある。`next.config.js` の `rewrites({ beforeFiles })` で回避が必要
- **改善策**: 動的ルートがあるプロジェクトで `public/` に新規ファイルを追加する際は、ルート競合を確認する。競合する場合は `rewrites` で明示的にルーティングする
- **影響度**: 中（ステージングデプロイ後にユーザーが検出・対応）
- **キーフレーズ**: 「動的ルートは静的ファイルを食う。[slug] がある場所に public/ ファイルを追加するなら rewrites で保護する」

### パターン24: ローカル開発環境なしでの変更
- **何がうまくいかなかったか**: `.env` 未設定でdevサーバーが起動せず、ローカル検証をスキップしたため P22, P23 がデプロイ後に発覚
- **根本原因**: 外部プロジェクトのローカル環境セットアップを事前確認しなかった
- **改善策**: `.env` がなくても `npm run build` でTypeScript型チェックは可能。最低限ビルド検証を試みる
- **影響度**: 中（デプロイ失敗→修正→再デプロイのサイクル）
- **キーフレーズ**: 「ローカルで検証できないなら、せめてビルドを通す」

### パターン25: 装飾用オーバーレイによるポインターイベントブロック（3x3 Hero CTAバグ）
- **何がうまくいかなかったか**: ヒーローセクションの装飾用グラデーションオーバーレイ（`absolute inset-x-0 bottom-0 h-1/3`）が、CTAボタンのクリックをブロックしていた。モバイルでのみ発生（デスクトップではボタンがグラデーション領域外）
- **根本原因**:
  1. 装飾用 absolute 要素に `pointer-events-none` を付与する基本パターンが適用されていなかった
  2. DOM順序でオーバーレイがコンテンツより後にあるため、ブラウザのスタッキングコンテキストでオーバーレイが上に来る
  3. デスクトップではヒーロー高さが十分でボタンとグラデーションが重ならないため、問題が見えなかった
- **改善策**: 装飾用の absolute/fixed 要素には必ず `pointer-events-none` を付与する。例外的にクリック可能にする場合のみ明示的に `pointer-events-auto` を設定する
- **影響度**: 大（モバイルユーザー全員がCTAを使用不能。本番ユーザーが発見）
- **キーフレーズ**: 「装飾要素は pointer-events-none。触れない要素は触れなくする」

### パターン26: モバイル固有バグのQA漏れ
- **何がうまくいかなかったか**: モバイルでのみ発生するCTAクリック不能バグがリリース前に検出されなかった
- **根本原因**:
  1. リリース前のQAに「モバイルビューポートで全CTA/リンクを実際にクリックする」項目がなかった
  2. レスポンシブ検証が「見た目の確認」に留まり、「操作の確認」が行われていなかった
  3. ビューポート依存の要素重なり（absolute要素とインタラクティブ要素）の確認が設計時に行われていなかった
- **改善策**: リリース前に最低375px幅で全CTA/リンクのクリック遷移を確認する。レスポンシブデザインでは「見た目」だけでなく「操作」を検証する
- **影響度**: 大（プロセスの欠陥。今後のプロジェクトにも横展開必要）
- **キーフレーズ**: 「レスポンシブQAは見た目だけでは不十分。押せるものは押して確認する」

## プロセステンプレート

### モバイルQAチェックリスト（リリース前必須）
```
1. ブラウザDevToolsでモバイルビューポート（375px幅）に設定
2. 全CTA/リンクボタンを実際にクリックし、遷移先が正しいことを確認
3. absolute/fixed 配置の装飾要素がインタラクティブ要素に重なっていないか確認
   - DevTools の Elements パネルで要素をホバーし、重なりを視覚的に確認
   - 疑わしい場合: 装飾要素に pointer-events-none があるか確認
4. フォーム入力欄のフォーカス・入力が可能か確認
5. スクロール操作が正常か確認（固定ヘッダー等にブロックされていないか）
6. タブレット（768px幅）でも同様の確認を実施
```

### 装飾オーバーレイの安全な実装パターン
```
装飾用 absolute/fixed 要素のチェック:
1. pointer-events-none を付与しているか？
2. インタラクティブ要素（リンク、ボタン、入力欄）と重なる可能性があるか？
3. ビューポートが縮小したときに重なりが発生しないか？

安全なパターン:
  <div class="relative">
    <div class="relative z-10">  <!-- コンテンツ: z-index で上に -->
      <a href="/link">Click me</a>
    </div>
    <div class="absolute inset-0 pointer-events-none">  <!-- 装飾: pointer-events-none -->
      <!-- グラデーション、背景エフェクト等 -->
    </div>
  </div>

危険なパターン（今回のバグ）:
  <div class="relative">
    <div>  <!-- コンテンツ -->
      <a href="/link">Click me</a>
    </div>
    <div class="absolute bottom-0 h-1/3">  <!-- 装飾: pointer-events-none なし! -->
      <!-- グラデーション → ボタンのクリックを奪う -->
    </div>
  </div>
```

### 動的ルート + 静的ファイル競合の確認手順
```
1. プロジェクトに [slug] / [...catchAll] 動的ルートがあるか確認
2. ある場合、追加するファイルのパスが動的ルートにマッチしないか確認
   - /llms.txt → app/[jobType] にマッチする可能性あり
   - /sitemap-static.xml → 拡張子が異なるためマッチしない
3. マッチする場合、next.config.js に rewrites を追加:
   rewrites() {
     return {
       beforeFiles: [
         { source: "/llms.txt", destination: "/llms.txt" },
       ],
     }
   }
```
