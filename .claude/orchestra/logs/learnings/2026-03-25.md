## 失敗パターンと根本原因

### パターン16: 不要なClient Component追加による環境破壊（最重大）
- **何がうまくいかなかったか**: ナビのアクティブ状態表示のためにDesktopNav Client Componentを新規追加したところ、`npm run dev --prefix` 環境でwebpackのモジュール解決が壊れ、全ページでランタイムエラーが発生
- **根本原因**:
  1. Server Componentの静的リンクで十分な要件に対し、Client Componentを追加するのは過剰設計
  2. `--prefix` 環境がwebpackのcwdに影響することを理解していなかった
- **改善策**: Client Component追加前に「Server Componentで実現可能か」を必ず検討する。`--prefix` 環境では新規Client Componentの追加に特に注意する
- **影響度**: 大（全ページ停止、デバッグに大幅な時間浪費）

### パターン17: エラー原因の切り分け遅延（重大）
- **何がうまくいかなかったか**: ランタイムエラー発生時、`.next`キャッシュ削除とサーバー再起動を6回以上繰り返したが、最初にすべき「自分の変更を戻して再現確認」をしなかった
- **根本原因**:
  1. 「エラーの原因は自分のコードか環境か」の切り分けを最初に行う手順が確立されていなかった
  2. 確証バイアス: 「自分のコードは正しいはず」→ 環境要因を先に疑った
- **改善策**: エラー発生時の最初のステップは「自分の変更を一旦戻す（git stash）」。同じアプローチを3回繰り返しても解決しなければ、別のアプローチに切り替える
- **影響度**: 大（デバッグ時間の大幅浪費）
- **キーフレーズ**: 「キャッシュクリアは祈りではない。原因切り分けが先」

### パターン18: エラーの外部要因への誤帰属
- **何がうまくいかなかったか**: 自分のコード変更が原因のエラーを「プレビューツールのキャッシュ問題」と誤判断した
- **根本原因**: 直近で行った自分の変更を疑う前に、環境やツールのせいにする傾向
- **改善策**: 「エラーが発生したとき、直近の自分の変更が最も可能性の高い原因である」というデフォルト仮説を持つ
- **影響度**: 中（正しい原因特定の遅延）

### パターン19: Next.js metadata APIの対応範囲を確認せずに実装（セッション2）
- **何がうまくいかなかったか**: `metadata.alternates.types` に `text/plain` + `llms.txt` を設定したが、linkタグが生成されなかった
- **根本原因**: Next.js metadata APIが任意のalternate typeをサポートしないことを事前に確認しなかった。「おそらく動くだろう」で実装を進めた
- **改善策**: フレームワークの抽象化APIを使う前に、公式ドキュメントで対応範囲（何ができて何ができないか）を確認する
- **影響度**: 小（デプロイ1回分の手戻り）
- **キーフレーズ**: 「フレームワークAPIの対応範囲を仮定で判断しない。ドキュメントで確認する」

### パターン20: デプロイ前のローカル検証省略（セッション2）
- **何がうまくいかなかったか**: metadata API設定後、ローカルでHTML出力を確認せずにデプロイし、本番で問題を発見した
- **根本原因**: 「静的な設定変更だからローカル確認は不要」という油断
- **改善策**: head/metadata変更時は、デプロイ前にローカルでcurlまたはDevToolsで出力HTMLを必ず確認する
- **影響度**: 小（再デプロイの手間）

## プロセステンプレート

### Next.js エラーデバッグの標準手順
```
1. エラー内容を正確に記録する（エラーメッセージ、スタックトレース、発生条件）
2. 直近の自分の変更を確認する（git diff）
3. 変更を一旦戻す（git stash）→ エラーが再現するか確認
   - 再現しない → 自分の変更が原因。差分を細かく分割して原因特定
   - 再現する → 環境要因。キャッシュクリア、依存の再インストール等を試す
4. 同じアプローチを3回繰り返しても解決しない場合、アプローチを変える
5. SSRとCSRを分離して確認する（curl でSSR出力確認）
```

### Next.js Client Component追加の判断基準
```
Client Componentが必要な条件（全て該当する場合のみ追加）:
- [ ] useState, useEffect, useRouter等のReact Hooksが必須
- [ ] Server Componentでは実現不可能であることを確認済み
- [ ] 静的HTML + CSS（hover等）で代替できないことを確認済み
- [ ] --prefix等の非標準実行環境での動作を検証済み

不要な場合の代替手段:
- ナビのアクティブ状態 → CSS :current-page or サーバーサイドでのクラス付与
- ホバーエフェクト → CSS :hover
- 静的リンク → <a> タグ（Server Componentで十分）
```

### AI向けSEO導入チェックリスト（セッション2）
```
AI検索対応の実装手順:
1. public/llms.txt を作成（サイト概要、主要コンテンツ一覧、対象読者）
2. public/llms-full.txt を作成（各ページの構造化説明、FAQ、ドメイン知識）
3. robots.ts でAIクローラーを明示許可（GPTBot, Claude-Web, Anthropic-AI, PerplexityBot等）
4. layout.tsx の <head> に <link rel="alternate" type="text/plain" title="LLMs.txt" href="/llms.txt"> を追加
   - 注意: Next.js metadata.alternates.types は任意typeに対応しないため、直接 <link> タグで記述する
5. デプロイ前にローカルで検証:
   - curl localhost:3000/llms.txt → 200 OK
   - curl localhost:3000/llms-full.txt → 200 OK
   - curl -s localhost:3000 | grep llms.txt → link タグの存在確認
6. デプロイ後に本番で同様の検証を実施
```

### Next.js metadata API の限界と代替手段（セッション2）
```
metadata API でサポートされないケース:
- metadata.alternates.types に任意の MIME type（text/plain等）を設定 → link タグが生成されない
- RSS/Atom 以外の alternate type は非対応

代替手段:
- layout.tsx の <html> 内の <head> タグに直接 <link> を記述する
- 例: <link rel="alternate" type="text/plain" title="LLMs.txt" href="/llms.txt" />

一般原則:
- フレームワークの抽象化APIが対応しない場合、低レベルな手段（直接HTML記述）にフォールバックする
- 抽象化層で「おそらく動く」と仮定せず、事前にドキュメントで対応範囲を確認する
```

### デバッグ時の3回ルール
```
同じアプローチ（キャッシュクリア、再起動等）を3回繰り返しても解決しない場合:
1. そのアプローチを中止する
2. 問題の原因仮説を見直す
3. 別のアプローチを試す（変更を戻す、最小再現環境を作る、等）

「同じことを繰り返して異なる結果を期待するのは狂気の定義である」
```

### AI向けコンテンツには英語版が必須（セッション3）
- **何がうまくいかなかったか**: 日本語サイトだからという理由でllms.txtを日本語のみで作成。ユーザーから「LLMは英語で処理する」と指摘され、英語版を追加した
- **なぜうまくいかなかったか**: AI向けコンテンツの設計を「人間向けSEO」の延長で考えた。LLMの学習データの大部分が英語であり、英語の方が推論精度が高いという特性を考慮しなかった
- **改善策**: AI向けコンテンツ（llms.txt、構造化データ等）は、サイトの言語に関わらず英語版を必ず用意する
- **影響度**: 中（セッション1追加分の工数）
- **キーフレーズ**: 「AI向けコンテンツのターゲットはLLM。人間の言語ではなくLLMの処理特性で設計する」

### ユーザーフィードバックによる英語版追加の迅速な対応（セッション3）
- **何がうまくいったか**: ユーザーの指摘を受け、英語版（llms-en.txt, llms-full-en.txt）を即日で作成・デプロイ・検証完了
- **なぜうまくいったか**: 日本語版の構造をテンプレートとして踏襲したため効率的だった。hrefLang属性の付与など多言語対応のベストプラクティスも適用
- **再利用条件**: 多言語対応時は、先に作成した言語版をテンプレートにして構造を統一する
