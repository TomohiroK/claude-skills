## 成功パターン

### YouTubeからの動画ID取得: curlとgrep -oEの組み合わせ
- **何がうまくいったか**: WebSearch/WebFetchではYouTubeの動的ページからIDを取得できない制約をすばやく把握し、`curl` + `grep -oE`（macOS対応拡張正規表現）で19本分のIDを安定取得した
- **なぜうまくいったか**: 動的レンダリングされるページのデータはブラウザ相当のフェッチでは取得できないという原則を理解し、HTMLソースから直接パターンマッチする手法に切り替えた
- **再利用条件**: YouTube等の動的ページから動画IDを抽出する際は `curl <url> | grep -oE '[a-zA-Z0-9_-]{11}' | sort -u` で候補を絞り込む

### TypeScript型拡張を起点とした関連コンポーネントの一括更新
- **何がうまくいったか**: `TeamCategory`型に`WT`を追加した時点でTypeScriptのビルドエラーが全箇所を指摘し、TeamCategoryBadge・TeamFilters・teams/page.tsxのバリデーションを漏れなく更新できた
- **なぜうまくいったか**: 型定義を変更すると型エラーが連鎖的に発生するTypeScriptの性質を活用し、修正箇所の網羅性をコンパイラが保証した
- **再利用条件**: enumやunion型を拡張する際は、型エラーを修正の手引きとして活用する。`grep`による影響箇所の事前確認と組み合わせると確実性が高まる

## 失敗パターンと根本原因

### パターン27: データ追加時のエンティティ作成フロー理解ミス（最重大）
- **何がうまくいかなかったか**: 「4月イベントにチームが参加している」という情報を追加する際、イベントの説明テキストにチーム名を直接記述した。ユーザーから「ルールが違います。新しいチームがそのイベントにいたら、そのチームを作ってリンクをイベントに貼るです」と指摘を受け、全て手戻りになった
- **根本原因**:
  1. **コードベース調査の省略**: `team.repository.ts`・`event.repository.ts`のデータ構造を確認せずに作業着手した。既存のparticipantsフィールドとTeamエンティティの関係が未確認のまま「テキスト追記」という表面的な対応を選んだ
  2. **ドメインモデルの未把握**: 「データを充実化する」=「テキストを書く」と短絡的に解釈した。Webアプリケーションにおいては「データを充実化する」=「エンティティを作成し、リレーションを設定する」が正しい
  3. **既存パターンの確認不足**: 既にEvent ID 3のparticipantsにチームがリンクされているケースがあり、それを参照すれば正しいフローが分かった
- **改善策**:
  - データ追加・更新作業の前に、必ず既存のリポジトリファイル（`*.repository.ts`等）のデータ構造とリレーション定義を確認する
  - 同種のデータが既に存在する場合（例: 既にparticipantsにリンクされたチーム）を見つけ、そのパターンを踏襲する
  - 「エンティティを新規作成してリンクを貼る」パターンと「既存フィールドにテキストを書く」パターンを混同しない
- **影響度**: 大（初期対応が全て手戻り。ユーザーへの追加確認コストが発生）
- **キーフレーズ**: 「データ追加の前にリポジトリ構造を読む。テキストではなくエンティティで考える」

### パターン28: macOS grep -P フラグ非対応
- **何がうまくいかなかったか**: Linux標準の `grep -P`（Perl互換正規表現）を使用しようとしたが、macOS（BSD grep）は`-P`フラグ非対応でエラー
- **根本原因**: macOS（BSD）とLinux（GNU）でgrepの対応フラグが異なることを作業前に意識していなかった
- **改善策**:
  - macOSでは `grep -E` または `grep -oE`（拡張正規表現）を使用する
  - Perl互換が必要な場合は `perl -nle 'print $1 if /pattern/(capture)/'` を使用する
  - `sed`や`awk`はmacOSでも概ね同様に動作するが、`-i`フラグの挙動が異なる点に注意（macOSでは `-i ''`が必要）
- **影響度**: 小（即座に代替手段で対応。機能的な影響なし）
- **キーフレーズ**: 「macOSのgrepは -P 非対応。-oE か perl -nle で代替する」

## プロセステンプレート

### データ追加・更新作業の事前確認フロー
```
1. 既存のリポジトリファイルを確認する
   - どのエンティティが存在するか（team, event, player等）
   - エンティティ間のリレーション（participantsフィールド等）はどう定義されているか
   - 同種のデータが既に存在する場合、どのパターンで実装されているか

2. 追加すべき操作を特定する
   - 新規エンティティ作成が必要か
   - 既存エンティティへのリレーション設定が必要か
   - フィールドへのテキスト追記で十分か

3. 既存パターンを踏襲して実装する
   - 既存の同種データの実装を参照し、同じ構造・命名規則で実装する

例: 3x3-portalでチームをイベントに追加する正しいフロー
  1. team.repository.ts を確認 → Team型とカテゴリを把握
  2. 既存チームのparticipantsリンクを確認 → id参照のパターンを把握
  3. 新チームを teams 配列に追加（id, name, category等を設定）
  4. event の participants フィールドにチームIDを追加
  5. TypeScript型エラーがない状態でビルド確認
```

### macOS / Linux コマンド互換チートシート
```
正規表現grep:
  Linux:  grep -P 'pattern'   ← macOSでは動かない
  macOS:  grep -E 'pattern'   ← 拡張正規表現
  両対応: perl -nle 'print if /pattern/'

キャプチャ抽出:
  Linux:  grep -oP '(?<=prefix)\w+'
  macOS:  grep -oE 'prefix\w+' | sed 's/prefix//'
  両対応: perl -nle 'print $1 if /prefix(\w+)/'

ファイル内文字列置換:
  Linux:  sed -i 's/old/new/g' file
  macOS:  sed -i '' 's/old/new/g' file  ← '' が必須
```
