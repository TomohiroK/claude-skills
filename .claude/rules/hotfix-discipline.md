# ホットフィクス（本番バグ修正）の規律

## 第一原則: 修正の副作用を全コードパスでトレースする

ガード条件の追加・コードブロックの移動は、**そのメソッドの全呼び出し元**で期待動作を検証する。
「この呼び出し元は大丈夫だろう」と推測で省略しない。

## ガード条件追加時の必須チェック

### 全呼び出し元のトレース
メソッドにガード（早期 return）を追加する場合:
1. そのメソッドの全呼び出し元を grep で特定する
2. 各呼び出し元で、ガードが true/false になるケースを列挙する
3. ガードによって既存の正常フローが壊れないことを確認する

```
修正対象: _isIntegrate() にガード追加
  ↓
呼び出し元1: update() → ガードで false → OK（これが修正目的）
呼び出し元2: approval() → ガードで false → NG（承認時も会計連動しなくなる）
  ↓
対策: approval() では approval_at を事前セットしてガードを通過させる
```

### 条件式に使う変数の状態マトリクス
ガード条件に使う変数が、各呼び出し元でどの値を取るかをマトリクスで整理する:

| 呼び出し元 | $approvalSteps | $purchase['approval_at'] | ガード結果 | 期待 |
|-----------|---------------|------------------------|-----------|------|
| update() approver未設定 | null | empty | return false | OK |
| update() approver設定済 | 2 | empty | 通過 | OK |
| approval() approver未設定 | null | セット済み | 通過 | OK |
| approval() approver設定済 | 2 | セット済み | 通過 | OK |

## コードブロック移動時の変数依存チェック

### if ブロック内で定義される変数の追跡
コードブロックの順序を変更する際:
1. 移動するブロック内で**定義**される変数を全て洗い出す
2. それらの変数が移動先以降で**参照**されていないか確認する
3. 条件分岐（if）内で定義される変数は、分岐に入らなかった場合に未定義になることを認識する

```php
// 危険パターン: $ledgers_base は if 内でのみ定義
if ($this->_isIntegrate($purchase)) {
    $ledgers_base = ...;  // ここでのみ定義
}
// ... 後続で $ledgers_base を参照 → if に入らなかった場合 undefined
$purchase->accountterm_id = $ledgers_base['accountterm_id'];  // エラー
```

## PHP 緩い比較のトラップ

### null/false/0 が絡む比較は厳密比較（===）を使う
PHP の `==`/`!=` は型強制を行う。以下のトラップに注意:

| 式 | 結果 | 理由 |
|----|------|------|
| `null == 2` | false | null は 0 に変換 → 0 == 2 → false |
| `false != 2` | true | false は 0 に変換 → 0 != 2 → true |
| `null == 0` | true | null は 0 に変換 → 0 == 0 → true |
| `"" == 0` | true | "" は 0 に変換 → 0 == 0 → true |

ガード条件で `==`/`!=` を使う場合、入力値が `null`/`false`/`0`/`""` になるケースを必ずトレースする。

## エラー連鎖の根本原因特定

### 複数の症状が出たら根本原因を探す
1つのバグが複数の症状として現れることがある。各症状を個別に修正するのではなく、根本原因を特定する。

```
根本原因: _isIntegrate() のガード漏れ（自動承認）
  ↓
症状1: approval_status が勝手に 1 になる
症状2: 換算レートメッセージの誤表示（承認済み前提の条件判定）
症状3: 仕訳未作成（exchange_rate 判定ミスで integrated_status=0）
症状4: 承認取消エラー（accountterm_id が null で会計期間チェック失敗）
```

根本原因（症状1）を修正すれば、症状2-4は発生条件がなくなる。ただし、根本修正だけでなく各症状のガード（防御的プログラミング）も必要。

## 発生事例（2026-04-02）

webfront2020 の PurchaseController で `_isIntegrate()` にガードを追加。update() からの呼び出しは修正目的通りだったが、approval() からの呼び出しでもガードが効き、`$ledgers_base` が未定義になり本番で Whoops エラー。全呼び出し元のトレースを事前に行っていれば防げた。
