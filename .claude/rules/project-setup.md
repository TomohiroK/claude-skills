# プロジェクト作業の事前確認

- 新しいプロジェクトに触れる際は、まずビルドツール設定を確認する（Trunk.toml, Cargo.toml, package.json, launch.json等）
- 類似名ディレクトリが存在する場合、作業対象を最初に確認する
- 静的ファイル配置時はビルドパイプラインのコピー設定を必ず確認する（Trunkのcopy-file/copy-dir等）

## データ追加・更新作業の事前確認（必須）

データを追加・更新する作業（チーム追加、イベント更新、記事作成等）では、テキストを書く前にリポジトリ構造を確認する。

1. **エンティティ定義の確認**: `*.repository.ts`、`types/domain.ts`等でエンティティの型とリレーションを把握する
2. **既存パターンの参照**: 同種のデータが既に存在する場合、その実装を見て同じ構造・命名規則を踏襲する
3. **操作の種類を特定する**: 「新規エンティティを作成してリレーションを設定する」か「既存フィールドにテキストを追記する」かを区別する

**禁止**: リポジトリ構造を確認せずに「とりあえずテキストを書く」対応をする。エンティティで管理されているデータはエンティティとして追加する。

発生事例（2026-03-26）: 3x3-portalでチームをイベントに追加する際、team.repository.tsのエンティティ構造を確認せず説明テキストにチーム名を直書きした。ユーザーから「チームを作ってリンクを貼る」が正しいフローと指摘を受け、全て手戻りになった。

## エラー発生時のデバッグ手順（必須）

1. エラー内容を正確に記録する（エラーメッセージ、スタックトレース、発生条件）
2. **最初に自分の変更を疑う**: `git diff` で直近の変更を確認する
3. **変更を一旦戻す**: `git stash` → エラーが再現するか確認する
   - 再現しない → 自分の変更が原因。差分を細かく分割して原因特定
   - 再現する → 環境要因。キャッシュクリア、依存の再インストール等
4. **3回ルール**: 同じアプローチ（キャッシュクリア、再起動等）を3回繰り返しても解決しない場合、そのアプローチを中止し、別のアプローチに切り替える

禁止事項:
- エラーの原因を環境・ツールのせいと決めつけること（確証バイアスの排除）
- キャッシュクリアと再起動の無限ループ（「キャッシュクリアは祈りではない」）
