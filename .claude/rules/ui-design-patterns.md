# UIデザインパターン標準ルール

Mobbinおよび主要アプリの調査に基づく、再現可能なUIデザインパターン集。
エージェントチーム（ui-designer, ux-researcher, frontend-developer 等）が参照する共通ルール。

---

## 1. オンボーディング

### パターン: プログレッシブオンボーディング（体験先行型）
- 登録を求める前にコア機能を体験させる（Duolingo、Canva方式）
- 3ステップ以内に「最初の成功体験」を届ける
- 情報入力は最低限（名前 + メールのみ）、残りは段階的に収集

```
[ステップ構成]
1. 価値提案（1画面、5秒で伝わる）
2. コア体験のミニデモ（インタラクティブ）
3. アカウント作成（ソーシャルログイン優先）
```

### パターン: パーミッションプライミング
- システムの権限ダイアログを出す前に、なぜ必要かを自前UIで説明する
- 「許可しない」を選んでも機能が使えるフォールバックを用意する

```
[構成要素]
- アイコン or イラスト（権限の用途を視覚化）
- 説明文（1-2文: 「あなたの○○のために△△が必要です」）
- 主CTAボタン（「許可する」）
- 副アクション（「あとで」— 閉じるではなくスキップ）
```

### 禁止事項
- アプリ起動直後に5画面以上のチュートリアルスライドを表示しない
- 「スキップ」のないオンボーディングを作らない
- パーミッション要求を起動直後にまとめて出さない（文脈に合わせて分散させる）

---

## 2. ナビゲーション

### パターン: ボトムタブバー（5項目以内）
- 主要機能を5つ以内に絞り、画面下部にタブで配置する
- アクティブタブはアイコン + ラベルで明示する（アイコンのみは非推奨）
- タブバーは常時表示（スクロールで隠さない）

```
[タブバー仕様]
- 高さ: 49pt（iOS標準）/ 56dp（Material）
- タッチターゲット: 最低 44×44pt
- アイコンサイズ: 24-28pt
- ラベルフォントサイズ: 10-12pt
- アクティブ色: ブランドプライマリカラー
- 非アクティブ色: グレー（コントラスト比 3:1 以上）
```

### パターン: ボトムシート（補助情報・フィルター用）
- 詳細情報やフィルター設定を画面下部からスライドで表示する
- ドラッグハンドル（グラバー）を必ず配置する
- 3段階の高さ（peek / half / full）でスナップさせる

```
[ボトムシート仕様]
- peek高さ: コンテンツの要約が見える最小高さ
- half高さ: 画面の50%
- full高さ: 画面の90%（ステータスバーは残す）
- 背景: 半透明オーバーレイ（タップで閉じる）
- ドラッグハンドル: 幅36pt × 高さ5pt、角丸、上部中央配置
```

### パターン: ジェスチャーナビゲーション
- 左スワイプで戻る（iOS標準。独自ジェスチャーで上書きしない）
- カード型UIではスワイプで操作（削除、アーカイブ等）
- ジェスチャーには必ずボタン代替を用意する（アクセシビリティ）

### 禁止事項
- ハンバーガーメニューを主要ナビゲーションにしない（発見率が低い）
- タブ6つ以上を配置しない
- ジェスチャーのみで完結する操作を作らない（ボタン代替必須）

---

## 3. 認証フロー

### パターン: ソーシャルログイン優先 + パスキー
- 「Apple でサインイン」「Google でサインイン」を最上位に配置する
- メールアドレス + パスワードは折りたたみ or 下部配置
- パスキー（FIDO2 / Face ID / Touch ID）対応を推進する

```
[認証画面の構成順序]
1. Apple / Google ボタン（プライマリ）
2. パスキー / 生体認証ボタン
3. 区切り線（「または」）
4. メール + パスワード（セカンダリ）
5. 「アカウントをお持ちの方 / 新規作成」の切り替えリンク
```

### パターン: メール列挙防止
- 「このメールアドレスは登録されていません」と表示しない
- ログイン失敗時は「メールアドレスまたはパスワードが正しくありません」で統一する
- パスワードリセット時も「登録されていればメールを送信しました」で統一する

### 仕様
- パスワード最低長: 8文字以上
- パスワード強度メーター: リアルタイム表示
- 入力フィールドのフォントサイズ: 最低16px（iOS自動ズーム防止）

---

## 4. 検索・フィルター

### パターン: インクリメンタル検索
- 入力と同時に候補を表示する（debounce: 300ms）
- 検索履歴・人気ワードをプレースホルダー下に表示する
- 検索バーをタップ時にフルスクリーン検索画面に遷移する

```
[検索画面構成]
- 検索バー（上部固定、オートフォーカス）
- 検索履歴（最新5件、クリアボタン付き）
- 人気検索ワード / トレンド
- インクリメンタル候補リスト
- 「該当なし」時のエンプティステート（提案・CTA付き）
```

### パターン: フィルターUI（ボトムシート型）
- フィルターはボトムシートまたは専用画面で提供する
- フィルター適用後の件数をリアルタイムで表示する（「123件を表示」）
- 「リセット」ボタンを必ず配置する

```
[フィルター仕様]
- 適用件数: フィルター変更のたびにリアルタイム更新
- チップ（タグ）: 適用中フィルターを検索結果上部に表示
- 各チップに「×」で個別解除可能
- 「すべてリセット」ボタン
```

### 禁止事項
- フィルター適用後の件数を表示しないフィルターUIを作らない
- 検索結果0件で何の説明もない画面を作らない

---

## 5. コンテンツ表示

### パターン: カード型レイアウト
- 1カードに1つの主題（情報を詰め込みすぎない）
- 画像 + タイトル + 補足テキスト + CTAの4要素構成
- カード間のスペーシング: 12-16pt

```
[カード仕様]
- 角丸: 12-16pt
- シャドウ: 控えめ（blur 4-8pt, opacity 0.05-0.1）
- パディング: 16pt
- 画像アスペクト比: 16:9 or 4:3 で統一
- タイトル: 最大2行（超過はellipsis）
- 補足テキスト: 最大3行
```

### パターン: リスト / グリッド切り替え
- ユーザーが表示形式を選べるトグルを提供する
- 選択状態はローカルに保持する（次回訪問時も維持）

### パターン: スケルトンスクリーン（ローディング）
- スピナーの代わりにコンテンツの骨格を表示する
- 実際のレイアウトと同じ形状のプレースホルダーを使う
- アニメーション: 左から右へのシマー（pulse）効果

```
[スケルトン仕様]
- 背景色: #E0E0E0（ライト）/ #3A3A3A（ダーク）
- アニメーション: linear-gradient + translateX（左→右、1.5秒周期）
- テキスト行: 高さ12-16pt、角丸4pt、幅はランダム（60-100%）
- 画像エリア: 実際の画像と同じアスペクト比
```

### 禁止事項
- 全画面スピナーのみでローディング状態を表現しない（スケルトン推奨）
- カードに5つ以上のアクションボタンを詰め込まない

---

## 6. フォーム設計

### パターン: インラインバリデーション
- フィールドからフォーカスが外れたタイミング（onBlur）で即時検証する
- 送信後にまとめてエラーを表示する方式は非推奨
- 成功時は緑チェック、エラー時は赤テキスト + アイコンで即時フィードバック

```
[バリデーション仕様]
- トリガー: onBlur（入力中は検証しない — ユーザーの思考を妨げる）
- エラー表示: フィールド直下、赤テキスト（#DC2626相当）+ エラーアイコン
- 成功表示: フィールド右端に緑チェックアイコン
- エラーメッセージ: 具体的（「8文字以上で入力してください」）、抽象的でない（「不正な入力」は不可）
```

### パターン: マルチステップフォーム（ウィザード）
- 長いフォームはステップに分割する（1ステップ3-5フィールド以内）
- プログレスバーまたはステップインジケーターを常時表示する
- 「戻る」で前のステップの入力値を保持する

```
[マルチステップ仕様]
- ステップ数: 最大5（それ以上は情報設計を見直す）
- プログレス表示: 「ステップ 2/4」+ 進捗バー
- 各ステップの見出し: ステップの目的を1行で説明
- 「次へ」ボタン: そのステップのバリデーション通過後にアクティブ化
- 「戻る」ボタン: 常に利用可能（入力値は保持）
```

### 仕様（共通）
- 入力フィールドのフォントサイズ: 最低16px（iOS自動ズーム防止）
- タッチターゲット: 最低 44×44pt
- ラベル: フィールド上部に常時表示（プレースホルダーをラベル代わりにしない）
- 必須/任意の明示: 少ない方にマーク（必須が多ければ「任意」にマーク）

### 禁止事項
- プレースホルダーテキストをラベルの代わりにしない（入力開始で消えて文脈を失う）
- 送信後に画面上部にスクロールしてエラーを表示する方式にしない（エラー箇所へスクロール）
- 入力中（onChange / onInput）にバリデーションエラーを表示しない

## 7. フィードバック・ステート

### パターン: トースト通知
- 一時的な成功/情報メッセージに使う（画面下部、3-5秒で自動消滅）
- エラーや取り消し不可の操作にはトーストを使わない（モーダル使用）

```
[トースト仕様]
- 位置: 画面下部、タブバーの上
- 表示時間: 3-5秒（アクション付きは5秒以上）
- 幅: 画面幅の90%（モバイル）
- 角丸: 8-12pt
- 種別（色）: success(緑), info(青), warning(黄), error(赤)
- 「取り消し」アクション: 可能な操作には必ず付与
```

### パターン: エンプティステート
- データが空の画面には必ずイラスト + 説明 + CTAを配置する
- 「データがありません」だけの画面は禁止

```
[エンプティステート構成]
- イラスト or アイコン（画面中央、64-120pt）
- メインメッセージ（1行、16-18pt、太字）
- 補足テキスト（1-2行、14pt、グレー）
- CTAボタン（主アクションを促す）

例:
  [イラスト: 空のブックマーク]
  「まだブックマークがありません」
  「気になった記事をブックマークして、あとで読みましょう」
  [ブックマークする記事を探す] ← CTAボタン
```

### パターン: モーダル（確認ダイアログ）
- 取り消し不可の操作（削除、送信、支払い等）の確認に使う
- 背景オーバーレイ + 中央配置
- 破壊的アクションのボタンは赤色にする

```
[モーダル使い分け基準]
- 情報通知（成功・完了）→ トースト
- 軽い警告・確認 → トースト + 取り消しアクション
- 取り消し不可の操作確認 → モーダル（明示的な確認ボタン必須）
- エラー（ユーザーの対処が必要）→ モーダルまたはインラインエラー
```

### 禁止事項
- エラー通知にトーストを使わない（見逃すリスクが高い）
- モーダルの中にモーダルを重ねない
- データが空の状態で説明もCTAもない画面を作らない

## 8. 設定・プロフィール

### パターン: グループ化設定リスト
- 設定項目をカテゴリごとにセクション分けする
- 各セクションにヘッダーラベルを付ける
- トグル / 遷移 / 値選択を視覚的に区別する

```
[設定画面構成]
- セクションヘッダー: 大文字 or グレーテキスト（12pt）
- 行の高さ: 44-48pt
- トグル: 右端に配置（即時反映、確認不要）
- 遷移（詳細画面へ）: 右端にシェブロン（>）
- 現在値: 右端にグレーテキストで表示
- 危険な操作（ログアウト、アカウント削除）: セクション最下部、赤テキスト
```

### パターン: プロフィール編集
- 編集可能フィールドを明示する（タップで編集モードに遷移）
- アバター画像: タップでカメラ/ライブラリ選択
- プロフィール充実度ゲージを表示する（入力促進）

## 9. 共通仕様（全パターン横断）

### タッチターゲット
- 最小サイズ: 44×44pt（Apple HIG）/ 48×48dp（Material Design）
- ボタン間の最小間隔: 8pt

### タイポグラフィスケール
```
[推奨スケール]
- 大見出し: 28-34pt / Bold
- 見出し: 22-24pt / Semibold
- 小見出し: 17-20pt / Semibold
- 本文: 16-17pt / Regular
- 補足テキスト: 13-14pt / Regular
- キャプション: 11-12pt / Regular
```

### スペーシングスケール
```
[推奨スケール（4ptベース）]
- 2xs: 4pt
- xs: 8pt
- sm: 12pt
- md: 16pt
- lg: 24pt
- xl: 32pt
- 2xl: 48pt
```

### ダークモード対応
- システム設定に追従する（アプリ独自のテーマ切り替えUIは非推奨）
- 色はセマンティックトークンで管理する（`background-primary`, `text-secondary` 等）
- 純黒（#000000）を背景に使わない（#1C1C1E 等のダークグレーを使用）

### アニメーション
- duration: 200-300ms（マイクロインタラクション）、300-500ms（画面遷移）
- easing: ease-out（登場）、ease-in（退場）、ease-in-out（移動）
- `prefers-reduced-motion: reduce` に対応する（アニメーション無効化）

### 単位の統一
- 本ドキュメントの pt / dp は全て **CSS px** として実装する（1pt ≒ 1px、1dp ≒ 1px）
- Tailwind CSS 使用時は rem 換算（1rem = 16px）で対応する

### アクセシビリティ（WCAG 2.1 AA 準拠を必須とする）
- コントラスト比: テキスト 4.5:1 以上、大テキスト 3:1 以上
- フォーカスインジケーター: 全インタラクティブ要素に視覚的なフォーカス表示
- スクリーンリーダー: 画像にalt、ボタンにラベル、ランドマークにrole
- キーボード操作: Tab順序が論理的、Escでモーダル閉じ

## 10. 2025-2026年トレンド（参考）

| トレンド | 概要 | 適用判断 |
|---------|------|---------|
| Liquid Glass（iOS 26） | すりガラス風の半透明UI | Apple HIG更新後に検討 |
| パスキー（FIDO2） | パスワードレス認証 | 新規認証フロー設計時に採用 |
| AI自然言語検索 | 検索バーにAI応答を統合 | AI機能を持つプロダクトで検討 |
| スケルトン→コンテンツフェード | スケルトンからの段階的表示 | 全ローディング状態に適用推奨 |
| ボトムシート中心設計 | モーダルよりボトムシートを優先 | 新規画面設計時に適用 |
