# セキュリティ監査（ペネトレーションテスト）ルール

## 事前準備（必須）

### 書面による許可の取得
- テスト実施前に対象システムの管理者から **書面** で許可を取得する（口頭のみは禁止）
- 書面には以下を明記する:
  - テスト対象のシステム・ドメイン・IPアドレス範囲
  - テスト期間（開始日〜終了日）
  - テスト手法の範囲（許可する手法・禁止する手法）
  - 本番環境への影響許容範囲（データ作成の可否、負荷上限等）
  - 発見した脆弱性の報告先・報告期限
  - レポートの保管・共有・廃棄のルール
  - 免責・損害賠償に関する合意

### クラウドプロバイダのポリシー確認
- AWS, GCP, Azure, Firebase等のペネトレーションテストポリシーを確認する
- 事前通知・申請が必要な場合はテスト前に完了させる

### テスト範囲の事前合意
- In-scope / Out-of-scope を明確にリスト化する
- 第三者サービス（認証基盤、CDN、決済等）は原則 Out-of-scope

## 標準チェック項目（6フェーズ）

### Phase 1: 偵察
- 技術スタック特定（レスポンスヘッダー、JSバンドル、エラーページ）
- フレームワークバージョン特定 → 既知CVE照合
- JSバンドル解析（APIエンドポイント、データ構造、環境変数）
- ビルドマニフェスト / ソースマップ露出確認
- DNS / サブドメイン列挙
- robots.txt / sitemap.xml 確認

### Phase 2: 構成検査
- セキュリティヘッダー7種（HSTS, CSP, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy, COEP/COOP）
- TLSバージョン（1.2以上のみ許可）
- CORS設定（Origin: null, 任意Origin, ワイルドカード）
- Cookie属性（Secure, HttpOnly, SameSite, Path, Domain）
- HTTPメソッド制限（全エンドポイント）
- 情報漏洩ヘッダー（x-powered-by, server等）
- キャッシュ制御（認証済みレスポンスの no-store 確認）

### Phase 3: 認証・認可
- サインアップ制限（メール確認必須か、ドメイン制限）
- パスワードポリシー（最低長、複雑性）
- メール列挙防止
- セッション管理（トークン有効期限、リフレッシュ、同時セッション）
- 権限昇格テスト（水平・垂直）
- CSRF保護（state-changing APIの全数検査）

### Phase 4: BaaS固有検査（Firebase等）
- セキュリティルール（Firestore、Realtime Database）
- ストレージバケット権限
- Cloud Functions の認証チェック
- APIキーのリファラー制限
- App Check の有効状態

### Phase 5: アプリケーション検査
- インジェクション（XSS, SQLi, NoSQLi, SSRF）
- パストラバーサル
- ファイルアップロード制限
- レート制限（全API、ログイン、パスワードリセット、フォーム送信）
- ビジネスロジック（価格改ざん、二重実行、レースコンディション）
- オープンリダイレクト

### Phase 6: サプライチェーン
- 依存パッケージの既知脆弱性
- サードパーティスクリプトのSRI（Subresource Integrity）
- CDN経由ライブラリの改ざん検知

## テスト中のルール
- テスト用アカウントのメールアドレスは `example.com`（RFC 2606予約ドメイン）を使用する
- テスト中に作成したデータは即座に削除する
- 本番データの閲覧・取得・変更は行わない
- 発見した脆弱性を悪用しない（実証は最小限の確認に留める）
- 合意範囲外のシステムへのアクセス禁止
- DoS/DDoS攻撃は明示的許可がない限り禁止

## テスト完了後

### レポートの取り扱い
- 保管: 暗号化またはアクセス制限必須。公開リポジトリへの配置は厳禁
- 共有: パスワード付きファイルまたはE2E暗号化チャネルのみ（平文メール禁止）
- 外部共有時は機密値（APIキー等）をマスキング
- 保管期間: 修正完了確認後90日以内に廃棄を推奨

### クリーンアップ
- テスト用データの削除確認
- 管理者にログ残存の確認・クリーンアップを依頼
- 個人情報に該当するデータが残存していないことを確認

## 実施タイミング
- 四半期: フル監査（Phase 1-6）
- 毎デプロイ: 自動チェック（Phase 2相当: ヘッダー、TLS、Cookie）
- イベント駆動: 認証方式変更、フレームワーク更新、決済機能変更時に即時実施

## CxO召集パターン
- セキュリティ監査: CTO, security-auditor, CLO, 記録係

## 法的根拠
- 不正アクセス行為の禁止等に関する法律（平成11年法律第128号）
- 個人情報の保護に関する法律（平成15年法律第57号）
- 各クラウドプロバイダの利用規約・ペネトレーションテストポリシー
