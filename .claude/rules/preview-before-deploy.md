# UI/CSS変更のプレビュー検証ルール

## 第一原則: 目で見て確認できないものをデプロイしない

CSS/UIの修正は、**人間の目（スクリーンショット）で確認してからデプロイする**。
HTMLテキストの確認（WebFetch、curl等）では視覚バグは検出できない。

## 必須フロー

UI/CSS変更を含むデプロイの前に、以下を**必ずこの順序で**実施する:

```
1. preview_start     — dev serverを起動する
2. preview_resize    — モバイル (375px) に設定する
3. preview_screenshot — スクリーンショットで視覚確認する
4. 問題なし確認      — 初めてデプロイ可能
```

### 省略不可
- 「1行のCSS変更だから」→ プレビュー必須
- 「前回のデプロイで確認済みだから」→ 今回の変更後にプレビュー必須
- 「構造は変えていないから」→ プレビュー必須

## WebFetch / curl での確認が不十分な理由

| 検出方法 | 検出可能 | 検出不可能 |
|---------|---------|-----------|
| WebFetch / curl | HTMLタグの有無、クラス名 | オーバーフロー、重なり、改行、余白崩れ |
| preview_snapshot | DOMツリー、テキスト内容 | 視覚的なレイアウト崩れ |
| **preview_screenshot** | **全て** | — |

CSS/UIの問題の大半は「視覚的なレイアウト崩れ」であり、テキストベースの確認では原理的に検出できない。

## hookが警告を出した場合

dev server起動を促すhookメッセージが表示された場合、**無視しない**。hookは過去の事故から学んだルールの自動適用である。

## 発生事例（2026-03-27）

イベント詳細ページのバッジ見切れバグで、プレビュー検証なしに4回デプロイを繰り返した。WebFetchでHTMLを確認しても視覚的なオーバーフローは検出できなかった。hookが「dev serverを起動しろ」と警告していたが無視し続けた。最初の1回でpreview_screenshotを撮っていれば、1回のデプロイで完了していた。
