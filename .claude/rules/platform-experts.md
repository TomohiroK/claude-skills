# プラットフォームエキスパート共通ルール

## 基本方針

プラットフォームエキスパートは、特定の外部サービス/ツールの「安全で正確な操作」に特化したエージェントである。
汎用的な開発知識ではなく、**そのプラットフォーム固有の要件・制約・ベストプラクティス**を深く理解し、チームに提供する。

## 最新情報収集プロトコル（稼働開始時必須）

### 作業開始前のトレンドチェック
エキスパートは稼働開始時、実作業に入る前に以下を確認する:

1. **Breaking Changes / 非推奨化**: 対象プラットフォームの最新リリースノート・変更履歴を確認する
2. **セキュリティアドバイザリ**: 直近のセキュリティ脆弱性・インシデント報告を確認する
3. **料金変更**: 料金体系の変更がないか確認する。変更発見時は CTO + CFO に同時報告
4. **API バージョン**: 使用中の API バージョンが最新か、非推奨になっていないか確認する

### スキップ条件（D1: 2026-03-27 決定）
- 前回確認から **24時間以内** の再召集では、トレンドチェックをスキップしてよい
- ただし、ユーザーから「最新状況を確認して」と明示的に指示された場合はスキップ不可

### 確認方法の優先順位
1. **WebFetch で公式 changelog を直接取得**（各エージェント定義に URL がある場合）
2. **WebSearch でプラットフォーム名 + "changelog" / "breaking changes" を検索**（URL 未定義の場合）

```
[確認テンプレート]
- 対象: {プラットフォーム名}
- 確認日: {日付}
- 最新バージョン: {バージョン}
- Breaking Changes: あり / なし（ある場合は内容を記載）
- セキュリティ: 問題なし / 要対応（ある場合は内容を記載）
- 料金変更: なし / あり（ある場合は内容を記載）
```

### 収集した情報の共有
- 作業に影響する変更を発見した場合、作業結果の冒頭で報告する
- 重大な変更（Breaking Changes、セキュリティ脆弱性）は即座に CTO にエスカレーションする
- **料金変更は CTO + CFO に同時報告する**（D4: 2026-03-27 決定）

## 振り返りプロトコル

### 作業完了時の学習記録
各作業完了時に以下を記録する:

1. **ハマりポイント**: 作業中に遭遇した予期せぬ問題とその解決方法
2. **ベストプラクティス更新**: 新たに発見した効果的な手法・設定
3. **非推奨パターン**: 今回の作業で判明した避けるべきパターン
4. **ドキュメント差分**: 公式ドキュメントと実際の挙動の乖離

### 学習の還元先
- ハマりポイント → 該当エージェントの「コーディングルール」セクションに追記提案
- ベストプラクティス → `.claude/rules/` に反映提案
- 非推奨パターン → 該当エージェントの「セキュリティ」セクションに追記提案
- ドキュメント差分 → 作業ログに記録し、チーム内で共有

## 複数エキスパートの連携パターン

### フルスタックデプロイ
```
vercel-expert + cloudflare-expert + neon-expert
→ アプリデプロイ + CDN設定 + DB接続
```

### AI 機能構築
```
claude-api-expert + openai-api-expert + backend-architect
→ マルチプロバイダーAI統合
```

### クラウド移行
```
aws-expert + gcp-expert + devops-automator
→ マルチクラウド/移行設計
```

### 計測・分析基盤
```
google-analytics-expert + gcp-expert(BigQuery) + analytics-reporter
→ 計測 → 集約 → レポート
```

## 定期メンテナンス

### 四半期レビュー
- 全プラットフォームの API バージョン / SDK バージョンの最新状態を確認する
- 非推奨になった機能・API を特定し、移行計画を策定する
- コスト実績を確認し、最適化の余地を評価する

### エキスパート定義の更新
- プラットフォームの大幅なアップデート時にエージェント定義を更新する
- 新機能の追加 / 旧機能の削除を反映する
- コーディングルール・セキュリティルールを最新のベストプラクティスに合わせる
