# クレデンシャル管理ルール

## 基本原則

1. **Claude 会話上でクレデンシャルを受け取らない** — API キー、パスワード、秘密鍵は Vault の inbox 経由で受け渡す
2. **ディスクに平文を置かない** — 暗号化済み（.age）ファイルのみ永続化する
3. **Vault は独立管理** — `service-account-manager` のみがアクセスする

## Vault 構成

```
~/.claude-vault/
├── master.key          # age 暗号鍵（600）— NEVER commit
├── accounts/           # 暗号化済み（*.age）
├── inbox/              # ユーザー → Vault の受け渡し場所
├── templates/          # サービス別テンプレート
├── tmp/                # 一時復号（自動削除）
├── logs/               # 監査ログ
└── vault.sh            # CLI ツール
```

## クレデンシャル登録フロー

```
ユーザー                     service-account-manager
  |                                |
  | 1. テンプレート要求            |
  |------------------------------->|
  |                                | vault.sh template <service>
  | 2. テンプレートを inbox/ に配置 |
  |<-------------------------------|
  |                                |
  | 3. テンプレートを編集           |
  | (直接ファイルを編集)            |
  |                                |
  | 4. 暗号化依頼                  |
  |------------------------------->|
  |                                | vault.sh inbox
  |                                | (暗号化 → 元ファイル安全削除)
  | 5. 完了通知                    |
  |<-------------------------------|
```

## プロジェクトへの注入フロー

```
エージェント（任意）          service-account-manager
  |                                |
  | 「プロジェクトXにDB接続が必要」 |
  |------------------------------->|
  |                                | vault.sh inject neon-db /path/to/project
  |                                | (.env に注入 → .gitignore 確認 → tmp 削除)
  | 「.env に注入完了」            |
  |<-------------------------------|
  |                                |
  | (.env を読んで作業)             |
```

## 全エージェント共通ルール

### 禁止事項
- クレデンシャルを会話に貼り付けるようユーザーに求めない
- `.env` ファイルの中身を Read して会話に出力しない
- クレデンシャルをコード内にハードコードしない
- クレデンシャルを含むファイルを git commit しない

### .gitignore 即時追加ルール（必須）

**クレデンシャルファイルをプロジェクトに配置した時点で、即座に `.gitignore` に追加する。ファイル配置と .gitignore 追加は不可分の1操作として扱う。**

対象ファイル:
- サービスアカウントキー（`serviceAccountKey.json`, `*-firebase-adminsdk-*.json` 等）
- `.env`, `.env.*`
- API キーを含むファイル（`credentials.json`, `*.pem`, `*.key` 等）
- OAuth クライアントシークレット

手順:
1. クレデンシャルファイルをプロジェクトにコピー/作成する
2. **同じ操作の中で** `.gitignore` にファイル名/パターンを追加する
3. `git status` で対象ファイルが追跡対象外であることを確認する

「後で .gitignore に追加する」は禁止。配置した瞬間に追加する。

#### 発生事例（2026-03-29）
kensyunavi プロジェクトで Firebase サービスアカウントキーを `serviceAccountKey.json` にリネームしてコピーしたが、`.gitignore` のパターン（`*-firebase-adminsdk-*.json`）に一致せず、GitHub にプッシュされた。GitHub 側でシークレット検出アラートが発生。リネーム後のファイル名も `.gitignore` に追加する必要があった。

### クレデンシャルが必要になった場合
1. `service-account-manager` に注入を依頼する
2. 注入された `.env` を参照する環境変数としてコードに組み込む
3. `.env` の中身を直接読んで出力しない

## 監査

- 全操作は `~/.claude-vault/logs/audit.log` に記録される
- 四半期ごとに監査ログをレビューする
- 不審な操作（大量の VIEW、未承認の INJECT 等）を検出した場合、CTO にエスカレーション

## 鍵ローテーション

- master.key のローテーション: 年1回
- サービスアカウントのローテーション: サービスの推奨期間に従う（通常90日）
- ローテーション手順: `vault.sh rotate <name>`
