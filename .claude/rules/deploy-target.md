# デプロイ先の確認ルール

## 第一原則: デプロイ先を確認せずにデプロイしない

デプロイは不可逆操作であり、間違った先にデプロイすると混乱を招く。
**デプロイ先（org/チーム/プロジェクト）が要件通りであることをユーザーに確認してからデプロイする。**

## 必須フロー

### デプロイ前の確認（省略不可）

1. **要件のデプロイ先を確認する**: ユーザーが指定した org / チーム / ドメインを把握する。ユーザーが既に指示済みであれば改めて聞き返さない
2. **現在の認証状態を確認する**: `vercel team ls` 等で、指定された org にアクセスできるか確認する
3. **アクセスできない場合は止まる**: 暫定的に別の org にデプロイしない。ユーザーに報告し、対応を仰ぐ
4. **指示通りのデプロイ先に実行する**: ユーザーの指示に従い、指定された org にデプロイする

### 禁止事項

| 禁止 | 理由 |
|------|------|
| 指定された org にアクセスできないとき、別の org に暫定デプロイする | ユーザーの意図と異なるデプロイ先に成果物が置かれる |
| デプロイ先を確認せずに `vercel deploy --prod` を実行する | 本番環境への不可逆操作を無確認で行うことになる |
| 認証エラーを自力で回避しようとする | org への招待・権限はユーザーにしか対応できない |

## 認証エラー時の対応

```
1. エラー内容をユーザーに報告する
2. 対応方法を提示する（vercel login、チーム招待の確認等）
3. ユーザーの指示を待つ
4. 自分で判断して別の場所にデプロイしない
```

## 発生事例（2026-04-01）

Bridge Note Indonesia コーポレートサイトのデプロイで、ユーザーが指定した `bridge-note` org にアクセスできなかった。ユーザーに確認せず `tomohiros-projects-47a483a4` に暫定デプロイした。ユーザーから「ディプロイ先を聞かないで作業すんな」と指摘。デプロイ先の変更は開発者の判断で行うべきではない。
