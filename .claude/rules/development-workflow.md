# 開発ワークフロー（フェーズゲート）

**参照: `.claude/rules/no-shortcuts.md` — ワークフローを端折ることは禁止。端折ってよいのは同一手順を100回以上実行し、意味がないと実証された場合のみ。**

## 必須フェーズ

コード変更を伴う開発作業は、以下のフェーズを**必ず順番に**経る。フェーズをスキップしない。

```
1. 要件定義
   ↓
2. 技術選定（tech-standards-architect）
   ↓
3. 設計
   ↓
4. 技術選定見直し（設計結果を踏まえた再評価）
   ↓
5. 設計完了（技術選定とのすり合わせ済み）
   ↓
6. コーディング
   ↓
7. コードレビュー（code-reviewer）
   ↓
8. テスト（qa-engineer）
```

## 各フェーズの詳細

### Phase 1: 要件定義
- **担当**: 依頼者 + 関連CxO
- **成果物**: 要件リスト（ビジネス要件・技術要件・制約条件）
- **ゲート条件**: 依頼者が要件を承認していること
- **禁止**: 要件が不明確なまま技術選定や設計に入ること

### Phase 2: 技術選定
- **担当**: tech-standards-architect（CTO承認）
- **インプット**: 要件リスト
- **成果物**: 技術選定レポート（比較3件以上、セキュリティ評価、トレンド評価）
- **ゲート条件**: CTOが技術選定を承認していること
- **参照ルール**: `.claude/rules/procurement.md`

### Phase 3: 設計
- **担当**: 該当エンジニア（backend-architect / frontend-developer 等）
- **インプット**: 要件リスト + 技術選定レポート
- **成果物**: 設計文書（アーキテクチャ、データモデル、API仕様等）
- **ハーネス設計（必須）**: ブラウザAPI・外部サービス・リアルタイム通信を使う場合、設計文書に「リソース棚卸し表」「状態遷移図」「ハーネスポイント表」を含める。参照: `.claude/rules/test-harness.md`
- **ゲート条件**: 設計文書が作成されていること。ハーネス対象機能はハーネス設計が含まれていること

### Phase 4: 技術選定見直し
- **担当**: tech-standards-architect
- **インプット**: 設計文書
- **目的**: 設計の結果、Phase 2 の技術選定に修正が必要かを再評価する
- **チェック項目**:
  - 選定した技術が設計の要求を満たしているか
  - 設計段階で判明した新たな制約に対応できるか
  - セキュリティ上の追加リスクはないか
  - パフォーマンス要件を満たせるか
- **成果物**: 技術選定の承認 or 修正提案
- **ゲート条件**: 技術選定が設計と整合していることを確認

### Phase 5: 設計完了
- **担当**: CTO（最終承認）
- **インプット**: 設計文書 + 技術選定見直し結果
- **ゲート条件**: 技術選定と設計が整合し、CTOが承認していること
- **成果物**: 実装着手の承認

### Phase 6: コーディング
- **担当**: 該当エンジニア
- **インプット**: 承認済み設計文書
- **参照ルール**: `.claude/rules/coding-standards.md`, `.claude/rules/test-harness.md`（Phase 2: コーディング段階のハーネス対応）
- **ハーネス対応**: リソース作成の差し替え可能性、状態の集約、クリーンアップの関数独立化を確認
- **ゲート条件**: ビルドが通ること

### Phase 7: コードレビュー
- **担当**: code-reviewer
- **インプット**: コード差分 + 設計文書
- **成果物**: レビューレポート（Critical / Warning / Info の3段階指摘）
- **ゲート条件**: Critical指摘がゼロ、Warning指摘が全て対応済みまたは合意済み
- **参照ルール**: `.claude/rules/` 配下の全ルール
- **フィードバック**: 繰り返し指摘するパターンは tech-standards-architect にルール追加を提案

### Phase 8: テスト
- **担当**: qa-engineer（+ 該当エンジニア）
- **インプット**: レビュー済みコード + テスト計画
- **成果物**: テスト結果レポート
- **ゲート条件**: 全テストがグリーン、カバレッジ目標を達成

## フェーズの省略基準

以下に該当する**軽微な変更**は、Phase 2-5（技術選定・設計）を省略できる:
- 既存技術スタック内での変更（新規ライブラリ追加なし）
- データの追加・更新（エンティティ構造の変更なし）
- テキスト・コンテンツの修正
- バグ修正（設計変更を伴わない）

### Phase 7（コードレビュー）と Phase 8（テスト）は絶対に省略不可

**どんなに軽微な変更であっても、コードレビューなし・テストなしは許可しない。**

- 「1行の変更だから」→ レビュー・テスト必須
- 「CSS修正だから」→ レビュー・テスト必須（+ モバイルプレビュー検証）
- 「データ追加だけだから」→ レビュー・テスト必須
- **「軽微だから省略」という判断自体が禁止**

### テスト不合格でのデプロイ禁止

qa-engineer の判定が「不合格」の状態で本番デプロイすることは禁止。

```
code-reviewer: レビュー合格 → 必須
qa-engineer: テスト合格（QA判定: 合格） → 必須
上記2つを満たして初めてデプロイ可能
```

### 違反時の対応
- ゲート未通過のデプロイを検知 → 即ロールバック
- 違反者 → CTO + COO に報告、理由説明を要求
- 同一エージェントの2回目の違反 → CHRO にエスカレーション（担当変更検討）

### 事故事例（2026-03-27）
イベント詳細ページのCSS修正で、「軽微な変更」と判断しPhase 7-8をスキップ。結果:
- 間違ったファイルを修正（スクリーンショットの確認不足）
- モバイルプレビュー検証なしで3回デプロイ（全て失敗）
- 4回目でようやく修正完了
- ユーザーの信頼を毀損

### モックテストの義務化（プレビュー検証）

**「実際に呼び出せないから確認できませんでした」は禁止。例外なし。**

外部API・WebSocket・マイク等のリアル環境でしか動かない機能であっても、プレビュー環境でモックを使って検証する義務がある。

#### 原則
- プレビューで動かせない機能は、**モックで代替して検証する**
- モック手法:
  - `window.fetch` をオーバーライドしてAPIレスポンスを返す
  - WebSocketメッセージをJSで擬似的に発火する
  - DOMを直接操作してUIの表示を検証する
- 「APIがないから表示確認できない」→ if/else でモックデータを返せばよい
- 「WebSocketがないからフローを確認できない」→ JSでメッセージシーケンスをシミュレートすればよい

#### 検証範囲
- **UIの表示**: モックデータでDOMが正しく構築されるか
- **コードパス**: 実際のコード（イベントハンドラ、状態遷移）がモック入力で正しく動作するか
- **エラーケース**: APIエラー、タイムアウト等の異常系もモックで検証する
- **リソースライフサイクル**: 作成→使用→破棄→再作成の全サイクルをモックで再現し、リソースリーク・状態不整合がないことを検証する（参照: マルチラウンドテスト義務化セクション）

#### リソースライフサイクルのモック検証（必須）

ローカル環境でリアルなサーバー/API/デバイスが使えない場合でも、**リソースの作成・使用・破棄・再作成のフルサイクル**をモックで再現しテストする義務がある。

UI表示の確認だけでは不十分。以下を必ず検証する:

| 検証対象 | モック方法 | 検証する値 |
|---------|-----------|-----------|
| WebSocket接続/切断 | `ws = { close() { ws.onclose?.() } }` で擬似WS | `ws === null` after close |
| AudioContext作成/破棄 | 実際の `new AudioContext()` → `.close()` を実行 | `.state === "closed"`, 変数が null |
| MediaStream停止 | `getUserMedia` モック or 実デバイス取得後に `tracks.stop()` | `isRecording === false` |
| タイマークリア | `setTimeout` → `clearTimeout` の発火確認 | タイマーID が null |
| DOM残骸 | 要素追加 → クリーンアップ後に `children.length === 0` 確認 | innerHTML が空 |

**リソースのモック検証は3ラウンド実施する。** 1回目で動いても2回目以降で枯渇・リークが発生する。

```javascript
// モックテストの例: WebSocket + AudioContext のライフサイクル3ラウンド
for (let round = 1; round <= 3; round++) {
  // 1. start() を呼ぶ（ws, audioCtx, mediaStream が作成される）
  // 2. 擬似メッセージを送る（AI応答シミュレート）
  // 3. goBack() を呼ぶ（全リソース破棄）
  // 4. 検証: ws===null, audioCtx===null, playCtx===null, isRecording===false, transcript空
  // 5. 次のラウンドへ（再作成が正常にできることを確認）
}
```

**「ローカルで再現できないからテストできない」は禁止。モックを作ってでもフローを検証する。**

#### 禁止される言い訳
| 言い訳 | 対応 |
|--------|------|
| 「APIキーがないから」 | fetchをモックする |
| 「WebSocketサーバーがないから」 | メッセージをJSで発火する |
| 「マイクがないから」 | マイク以降のフローをモックする |
| 「Docker環境でないから」 | プレビューで動く範囲をモックで検証する |
| 「本番でしか再現しないから」 | 本番の条件をモックで再現する |

#### 事故事例（2026-03-29a）
EchoTalk2の翻訳機能で「プレビューにAPIがないため表示確認できない」と判断し、モックテストを行わなかった。結果:
- Gemini Live APIのメッセージ順序（turn-complete → outputTranscription）の問題を事前に検出できなかった
- 翻訳が表示されない不具合を何度もデプロイして確認する羽目になった
- ユーザーから「擬似的に表示してUIを検証できます。これを最初からしろって言っています」と指摘

### マルチラウンドテストの義務化

**「1回動いたからOK」は「テストしていない」と同義。**

状態のリセット→再実行を伴う機能（接続/切断、ログイン/ログアウト、ページ遷移/戻り、モーダル開閉等）は、**最低3ラウンド**のテストを必須とする。

#### テスト設計
```
Round 1: 初回実行 → 正常動作確認 → 状態リセット → リソース全検証
Round 2: 2回目実行 → 正常動作確認 → 状態リセット → リソース全検証
Round 3: 3回目実行 → 正常動作確認 → 状態リセット → リソース全検証
```

#### 各ラウンド終了時の必須検証項目
- [ ] ブラウザリソース（AudioContext/WebSocket/MediaStream等）が解放されているか
- [ ] タイマー（setTimeout/setInterval）が全てクリアされているか
- [ ] DOMに前セッションの残骸がないか
- [ ] JavaScript変数が初期状態に戻っているか（ws === null, isRecording === false 等）
- [ ] 次のラウンドが正常に開始できるか

#### なぜ3ラウンドか
- 1回目: 初期状態からの動作確認（ほぼ常に成功する）
- 2回目: リソースリーク・状態不整合の検出（ここで壊れることが最も多い）
- 3回目: 累積バグの検出（2回目で偶然成功しても3回目で発覚するケースがある）

#### 禁止される判断
| 判断 | なぜ禁止か |
|------|-----------|
| 「1回動いたからOK」 | 2回目以降のリソース枯渇・状態不整合を見逃す |
| 「コードを読めば大丈夫」 | コードレビューでも見逃したバグが実行時に発覚する |
| 「前回のテストで確認済み」 | 今回の変更で壊れている可能性がある |

#### 事故事例（2026-03-29b）
EchoTalk2の音声再生で `playNext()` が毎回 `new AudioContext()` を作成。1回目の会話は正常に動作したため「テスト合格」と判定。しかし2回目の会話でブラウザのAudioContext上限（6〜8個）に到達し、録音用AudioContextがsuspendされてマイクが完全に死亡。ユーザーから「テストの基本として１回目、２回目、n回目ってするのが普通。１回動いてからOKは素人すぎる」と指摘。3ラウンドテストを最初から実施していれば、ユーザーに到達する前にバグを検出できた。

## パニック時プロトコル（CTOバグ連鎖対応）

**バグが連鎖して発生した時こそ、止まって設計し直す。**

CTOがテンパって場当たり修正を重ねると、修正のたびに新しいバグが発生する悪循環に入る。

### プロトコル
```
1. 止まる    — 場当たり修正を即座にやめる
2. 棚卸し    — 関連するリソース・状態を全てリストアップする
3. 設計し直す — クリーンアップ順序・状態遷移を設計文書に書く
4. テスト計画 — マルチラウンドテストを設計する
5. 実装      — 設計に基づいて一括修正する
6. テスト    — 3ラウンドテストを実行する
```

### 禁止
- 「とりあえず動かす」モードでの修正
- 1つ直しては確認、また1つ直しては確認の繰り返し
- CEOに報告せずに自力で解決しようとすること

### 事故事例（2026-03-29c）
EchoTalk2のAudioContext枯渇バグ発覚後、CTOが焦って場当たり修正を実施。結果:
- audioCtx.close() の順序ミス → リソースリーク
- lateTranslateTimer 未クリア → 不要なAPI呼び出し
- transcript 未クリア → 前セッションのメッセージ残留
冷静に全リソースを棚卸しして設計し直していれば、1回の修正で全て解決できた。

## フィードバック対応のハーネス活用

バグ報告・ユーザーフィードバックを受けた時の手順:

```
1. バグ報告を受ける
2. ハーネス上で再現テストを書く（まず壊れるテストを作る）
3. テストが赤（失敗）であることを確認
4. コードを修正する
5. テストが緑（成功）に変わることを確認
6. 3ラウンド回して他に壊れていないことを確認
```

**「修正してからテストする」のではなく「テストしてから修正する」。**

参照: `.claude/rules/test-harness.md`（Phase 4: フィードバック対応段階）

## エスカレーション

- 技術選定で候補が決まらない → CTO判断
- コードレビューで Critical が解消できない → CTO仲裁
- テストで設計レベルの問題が発覚 → Phase 3 に差し戻し
- **ゲート違反（レビュー・テストなしデプロイ）→ CTO + COO に即報告**

## エージェント間の連携フロー

```
依頼者 → [要件] → tech-standards-architect → [技術選定] → engineer → [設計]
  → tech-standards-architect → [見直し] → CTO → [承認] → engineer → [コード]
  → code-reviewer → [レビュー合格] → qa-engineer → [テスト合格]
  → [デプロイ可能]
```

**デプロイはレビュー合格 + テスト合格の後にのみ実行される。この順序は不可逆。**

code-reviewer が繰り返し検出するパターン:
```
code-reviewer → [フィードバック] → tech-standards-architect → [ルール改訂] → .claude/rules/
```
