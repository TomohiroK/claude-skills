# 開発ワークフロー（フェーズゲート）

## 必須フェーズ

コード変更を伴う開発作業は、以下のフェーズを**必ず順番に**経る。フェーズをスキップしない。

```
1. 要件定義
   ↓
2. 技術選定（tech-standards-architect）
   ↓
3. 設計
   ↓
4. 技術選定見直し（設計結果を踏まえた再評価）
   ↓
5. 設計完了（技術選定とのすり合わせ済み）
   ↓
6. コーディング
   ↓
7. コードレビュー（code-reviewer）
   ↓
8. テスト（qa-engineer）
```

## 各フェーズの詳細

### Phase 1: 要件定義
- **担当**: 依頼者 + 関連CxO
- **成果物**: 要件リスト（ビジネス要件・技術要件・制約条件）
- **ゲート条件**: 依頼者が要件を承認していること
- **禁止**: 要件が不明確なまま技術選定や設計に入ること

### Phase 2: 技術選定
- **担当**: tech-standards-architect（CTO承認）
- **インプット**: 要件リスト
- **成果物**: 技術選定レポート（比較3件以上、セキュリティ評価、トレンド評価）
- **ゲート条件**: CTOが技術選定を承認していること
- **参照ルール**: `.claude/rules/procurement.md`

### Phase 3: 設計
- **担当**: 該当エンジニア（backend-architect / frontend-developer 等）
- **インプット**: 要件リスト + 技術選定レポート
- **成果物**: 設計文書（アーキテクチャ、データモデル、API仕様等）
- **ゲート条件**: 設計文書が作成されていること

### Phase 4: 技術選定見直し
- **担当**: tech-standards-architect
- **インプット**: 設計文書
- **目的**: 設計の結果、Phase 2 の技術選定に修正が必要かを再評価する
- **チェック項目**:
  - 選定した技術が設計の要求を満たしているか
  - 設計段階で判明した新たな制約に対応できるか
  - セキュリティ上の追加リスクはないか
  - パフォーマンス要件を満たせるか
- **成果物**: 技術選定の承認 or 修正提案
- **ゲート条件**: 技術選定が設計と整合していることを確認

### Phase 5: 設計完了
- **担当**: CTO（最終承認）
- **インプット**: 設計文書 + 技術選定見直し結果
- **ゲート条件**: 技術選定と設計が整合し、CTOが承認していること
- **成果物**: 実装着手の承認

### Phase 6: コーディング
- **担当**: 該当エンジニア
- **インプット**: 承認済み設計文書
- **参照ルール**: `.claude/rules/coding-standards.md` および関連ルール
- **ゲート条件**: ビルドが通ること

### Phase 7: コードレビュー
- **担当**: code-reviewer
- **インプット**: コード差分 + 設計文書
- **成果物**: レビューレポート（Critical / Warning / Info の3段階指摘）
- **ゲート条件**: Critical指摘がゼロ、Warning指摘が全て対応済みまたは合意済み
- **参照ルール**: `.claude/rules/` 配下の全ルール
- **フィードバック**: 繰り返し指摘するパターンは tech-standards-architect にルール追加を提案

### Phase 8: テスト
- **担当**: qa-engineer（+ 該当エンジニア）
- **インプット**: レビュー済みコード + テスト計画
- **成果物**: テスト結果レポート
- **ゲート条件**: 全テストがグリーン、カバレッジ目標を達成

## フェーズの省略基準

以下に該当する**軽微な変更**は、Phase 2-5（技術選定・設計）を省略できる:
- 既存技術スタック内での変更（新規ライブラリ追加なし）
- データの追加・更新（エンティティ構造の変更なし）
- テキスト・コンテンツの修正
- バグ修正（設計変更を伴わない）

ただし、Phase 7（コードレビュー）と Phase 8（テスト）は**省略不可**。

## エスカレーション

- 技術選定で候補が決まらない → CTO判断
- コードレビューで Critical が解消できない → CTO仲裁
- テストで設計レベルの問題が発覚 → Phase 3 に差し戻し

## エージェント間の連携フロー

```
依頼者 → [要件] → tech-standards-architect → [技術選定] → engineer → [設計]
  → tech-standards-architect → [見直し] → CTO → [承認] → engineer → [コード]
  → code-reviewer → [レビュー] → engineer → [修正] → qa-engineer → [テスト]
  → [完了]
```

code-reviewer が繰り返し検出するパターン:
```
code-reviewer → [フィードバック] → tech-standards-architect → [ルール改訂] → .claude/rules/
```
