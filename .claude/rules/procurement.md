# プロダクト・ライブラリ・SaaS 導入ルール

## 基本方針

外部プロダクト・ライブラリ・SaaSを導入する際、**費用面と安全面の両方**を検討する。
片方だけの評価で意思決定しない。

## 費用面の検討

### コスト構造の全体把握
- 見えるコストだけで判断しない。以下の全層を算出する:

| コスト区分 | 確認項目 |
|-----------|---------|
| **初期費用** | セットアップ料、移行コスト、インテグレーション開発工数 |
| **月額/年額** | サブスクリプション料金、ライセンス料、従量課金の単価 |
| **従量課金の上振れ** | API呼び出し数、ストレージ量、帯域幅、ユーザー数に基づく最悪ケース試算 |
| **隠れたコスト** | サポート料、トレーニング、追加モジュール、データエクスポート料金 |
| **撤退コスト** | データ移行、代替サービスへの切り替え開発工数、契約解除違約金 |

### 料金プランの精査
- 無料プランの制約を明確にする（機能制限、レート制限、データ保持期間、SLA なし等）
- 次のプラン閾値を超えたときの料金ジャンプを確認する
- 年額 vs 月額の差額を確認する（年額のほうが安い場合でもロックイン期間を考慮）

### 従量課金のアラート設定
- 従量課金サービスには必ず予算アラートを設定する
- 月額予算の 50% / 80% / 100% で段階通知
- 上限設定（ハードリミット）が可能なサービスでは必ず設定する

### コスト比較の記録
- 導入時に最低3サービスの比較表を作成し、`docs/` に残す
- 比較項目: 機能、料金、SLA、セキュリティ、撤退コスト

```markdown
## サービス比較: 認証基盤（2026-03）
| 項目 | Firebase Auth | Auth0 | Clerk |
|------|-------------|-------|-------|
| 無料枠 | 50K MAU | 7.5K MAU | 10K MAU |
| 有料単価 | $0.01/MAU | $0.07/MAU | $0.02/MAU |
| 月1万MAU想定 | $0 | $23 | $0 |
| 月10万MAU想定 | $500 | $6,500 | $1,800 |
| SLA | 99.95% | 99.99% | 99.99% |
| 選定理由 | ... |
```

## 安全面の検討

### セキュリティ評価
- 導入前に以下を確認する:

| # | 確認項目 | 不合格基準 |
|---|---------|-----------|
| 1 | データの暗号化（転送中 + 保管中） | TLS なし or 保管時暗号化なし |
| 2 | 認証方式（API キー、OAuth、SAML） | 共有パスワードのみ |
| 3 | 権限の最小化（必要最低限のスコープ） | 全権限を要求 |
| 4 | SOC 2 / ISO 27001 等の認証 | 規模に応じて要求。小規模OSSは除外 |
| 5 | データの所在地（リージョン） | 日本/ASEAN に保管先がない場合は要検討 |
| 6 | インシデント対応ポリシー | 漏洩時の通知義務・対応フローが不明 |
| 7 | データポータビリティ | エクスポート不可 or プロプライエタリ形式のみ |

### OSSライブラリの安全評価
- npm / crate / pip パッケージの導入前に確認する:

| # | 確認項目 | 警戒基準 |
|---|---------|---------|
| 1 | メンテナンス状況 | 最終コミット1年以上前 |
| 2 | ダウンロード数 / GitHub Stars | 週100DL未満 or Stars 50未満 |
| 3 | 既知の脆弱性 | `npm audit` / `cargo audit` / `pip-audit` で検出あり |
| 4 | 依存の深さ | transitive dependencies が100以上 |
| 5 | ライセンス | GPL系（プロプライエタリ製品への組み込み制限）、ライセンス不明 |
| 6 | メンテナー数 | 単独メンテナー（バスファクター1）|

### SaaS連携時のデータフロー確認
- 自社からSaaSに送信するデータの一覧を明示する
- 個人情報・機密情報が含まれる場合、CLO（法務）を参加させる
- データの保持・削除ポリシーを確認する（退会後にデータが残るか）

```markdown
## データフロー: Gunplar → ImageKit
| データ項目 | 個人情報 | 機密 | 送信先リージョン |
|-----------|---------|------|----------------|
| 商品画像 | No | No | US (AWS) |
| ファイル名 | No | No | US (AWS) |
| ユーザーアバター | Yes | No | US (AWS) → 要検討 |
```

## ベンダーロックインの回避

### 標準規格を優先する
- プロプライエタリなAPIより、標準プロトコル（REST, GraphQL, OpenID Connect等）を採用しているサービスを優先
- データ形式はオープン標準（JSON, CSV, SQL dump等）でエクスポート可能であること

### 抽象化層の設計
- SaaS連携はラッパー層を挟む（`api-sdk.md` 参照）
- ラッパーのインターフェースをサービス非依存にし、差し替え可能にする

### 撤退計画の事前策定
- 導入時に「このサービスを辞める場合の手順」を `docs/` に残す
- データエクスポート手順、代替サービス候補、移行工数の概算を含める

## 承認フロー

### 費用発生の閾値別承認

| 月額コスト | 承認者 |
|-----------|-------|
| 無料 | 担当者判断（上記チェックリストの実施は必須）|
| ～$50/月 | CTO承認 |
| ～$500/月 | CTO + CFO承認 |
| $500超/月 | CTO + CFO + CEO承認 |

### 承認時の提出物
- サービス比較表（最低3候補）
- コスト試算（現在 + 12ヶ月後 + スケール時の最悪ケース）
- セキュリティチェックリスト結果
- データフロー図
- 撤退計画

## 定期レビュー

### 四半期コストレビュー
- 全SaaS/マネージドサービスの月額実績を集計し、予算との乖離を確認する
- 使用量が無料枠の80%を超えたサービスはアラート対象
- 利用実績がゼロ or 極小のサービスは解約を検討する

### 年次セキュリティレビュー
- 全連携サービスのセキュリティ認証（SOC2等）の有効期限を確認する
- 新たに報告された脆弱性・インシデントを確認する
- OSSライブラリの `audit` を実行し、脆弱性がないことを確認する
