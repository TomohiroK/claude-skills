# 顧客起点の設計ルール

## 第一原則: 顧客の課題から始める

マーケティング・SEO・コンテンツ制作・サイト設計は、技術的な実装から始めない。
**「誰が」「何に困って」「どう探すか」** を最初に定義する。

## 必須フロー（SEO・マーケティング・コンテンツ制作時）

### Step 1: ターゲット顧客の定義
- 業種、規模、役職、地域を特定する
- 「全員」はターゲットではない。具体的な人物像を描く

### Step 2: 顧客の悩み・課題の洗い出し
- 顧客が実際に抱えている課題を具体的な言葉で列挙する
- 例: 「税務相談したい」「会計をお願いしたい」「インドネシアに進出したい」「現地のことがわからない」
- 製品の機能ではなく、顧客の声の言葉で書く

### Step 3: 市場調査によるキーワード発見
- キーワードは自分たちで「決める」ものではない。**市場を調査して「発見する」**もの
- 顧客がそのサービスに求めるもの（安全？安心？安い？包括的？速い？）は調査して初めてわかる
- 調査手法:
  - 競合サイトのキーワード分析（どんな言葉で上位表示されているか）
  - 検索サジェスト・関連検索の調査（顧客が実際に入力している言葉）
  - 検索ボリュームと競合度の調査
  - ターゲット言語ごとに実施する（日本語、英語、インドネシア語等）
- **推測でキーワードを決めることは禁止**。データに基づいて発見する

### Step 4: コンテンツ・サイト構造の設計
- 各キーワードに対応するページ/セクションを設計する
- 顧客の悩みに直接答える構造にする（機能リストの羅列ではなく課題解決型）
- title / description / 見出しにキーワードを自然に含める

### Step 5: 仮説検証サイクルの設計
- マーケティングは **仮説 → 検証 → 仮説 → 検証** の繰り返し。一発で正解は出ない
- キーワード・訴求軸・CTA文言に対立する仮説がある場合、A/Bテスト用に複数パターンを用意する
- 設計段階で以下を組み込む:
  - **差し替えやすい構造設計**: 訴求文言・CTA・ヒーローコピー・キーワード等をコードに直書きせず、データファイルや定数に集約する。差し替え = 1ファイル編集で完了する構造にする
  - **テスト可能な構造**: A/Bパターンを切り替えられる仕組み（環境変数、クエリパラメータ、Cookie等）を設計に含める
  - **計測基盤**: どのパターンが成果を出しているか測定できるようにする（GA4イベント、コンバージョン計測等）
  - **判定基準**: 何をもって勝ちパターンとするかを事前に定義する（CTR、CVR、滞在時間等）
- 「安全」vs「安心」vs「安い」で迷うなら、3パターン作って検証する。議論で決めない、データで決める

### 差し替えやすさの設計指針
- **訴求コピー・キーワード**: content/ や lib/constants.ts 等のデータ層に集約。ページコンポーネントに直書きしない
- **CTA文言・ボタンテキスト**: 定数化して1箇所で管理。全ページのCTAを一括変更できる状態にする
- **ヒーロー画像・OGP**: パスを定数化。画像差し替え = ファイル上書きで完了
- **メタデータ（title / description）**: キーワード変更時に全ページ一括で追従できる構造にする
- **原則: 仮説検証のたびにコードを書き換える設計は失格。データの差し替えだけで済む設計が合格**

### Step 6: 技術的SEO
- robots.txt, sitemap.xml, JSON-LD, OGP 等の技術実装
- **ここが最後。Step 1-5 が終わってから実施する**

## 禁止事項

| 禁止 | 理由 |
|------|------|
| キーワード設計なしで title / description を書く | 顧客の検索意図と合致しないメタデータは無意味 |
| 機能リストを並べただけのページ | 顧客は「何ができるか」ではなく「自分の問題が解決するか」を知りたい |
| 技術的SEOだけ実装して「SEO対策完了」 | robots.txt と sitemap は土台。核心はキーワードとコンテンツ |
| ターゲットを定義せずにコンテンツを作る | 誰に向けて書いているか不明なコンテンツは誰にも刺さらない |
| 推測でキーワードを決める | 「安全」「安心」「安い」「包括的」のどれが刺さるかは市場調査で初めてわかる。自分の頭で考えて決めるものではない |

## 顧客課題型コンテンツの例

### Bad: 機能リスト型
```
Bridge Note の機能:
- 会計モジュール
- 売掛金管理
- 買掛金管理
- 固定資産管理
```

### Good: 課題解決型
```
インドネシアでの会計処理でお困りですか？

「現地の税務規制がわからない」
→ インドネシア税務に対応した固定資産管理と税務申告レポートを標準装備

「日本語で使えるインドネシアの会計ソフトがない」
→ 日本語・インドネシア語を含む5言語対応。日本人駐在員もローカルスタッフも同じシステムで作業可能

「複数拠点の会計を一元管理したい」
→ 1アカウントで複数法人を管理。権限設定でセキュリティも確保
```

## 発生事例（2026-04-02）

Bridge Note Indonesia のコーポレートサイトで「SEO対策をしろ」と依頼された際、robots.txt / sitemap.xml / JSON-LD / llms.txt 等の技術的チェックリストを埋めただけで完了とした。キーワード設計もターゲット定義もなし。ユーザーから「テキストを使え、SEO対策をしろ、やりましたじゃなくて、SEO対策するなら顧客の思考、顧客の悩みってなんだ？そもそも誰をターゲットにしている？って根本を考えられるようにしてください」と指摘。

## マーケティングチームの行動原則

1. **一発で正解を出そうとしない** — 仮説を立て、検証し、次の仮説を立てるサイクルを回す
2. **対立する仮説はテストで決着させる** — 会議室で議論しない。A/Bテストでデータを取る
3. **テスト可能な設計を最初から組み込む** — 後からA/Bテストを入れるのはコストが高い。設計段階で差し替え可能にする
4. **計測なきマーケティングは占い** — 全施策に計測ポイントを設定する
