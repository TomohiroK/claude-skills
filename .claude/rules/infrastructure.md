# インフラストラクチャ設計ルール

## 第一原則: オブザーバビリティ・ファースト

インフラ構成の選定時、「障害発生時にデバッグできるか」を最優先基準とする。
高機能・低コストよりも、ログ・メトリクス・トレースが取れる構成を選ぶ。

## エラーデバッグ基盤

### 構造化ログの必須化
- 全サービス/インスタンスでJSON構造化ログを出力する
- 最低限のフィールド: `timestamp`, `level`, `service`, `requestId`, `message`, `error`
- 自由文のログ（`echo "something happened"`）は禁止

```json
{
  "timestamp": "2026-03-25T10:30:00Z",
  "level": "error",
  "service": "api-server",
  "requestId": "req-abc123",
  "message": "Failed to query user",
  "error": {
    "type": "ConnectionTimeout",
    "message": "Connection to db-primary timed out after 5000ms",
    "stack": "..."
  },
  "context": {
    "userId": "user-456",
    "dbHost": "db-primary.internal",
    "queryDurationMs": 5000
  }
}
```

### リクエストIDの伝播
- リクエストの入口で一意のIDを採番し、全ての下流サービス・DBクエリ・外部API呼び出しに伝播する
- ロードバランサー → アプリサーバー → DB → 外部API の全区間でIDが追跡可能な状態にする
- ヘッダー名の標準: `X-Request-Id` または `traceparent`（W3C Trace Context）

### エラーの分類とアラート
- エラーを3段階に分類し、アラートレベルを分ける:
  - **Fatal**: サービス停止レベル → 即時通知（PagerDuty/Slack/電話）
  - **Error**: 機能障害だがサービスは継続 → 5分以内に通知
  - **Warning**: 性能劣化・閾値接近 → 日次サマリーで通知
- 「全エラーを同じ重要度で通知」はアラート疲れを招く。分類を必ず設計する

### DB エラーログ
- スロークエリログを有効化する（閾値: 100ms 以上）
- デッドロック検出ログを有効化する
- 接続プールの枯渇をログ出力する（available/waiting/max の3値）

## パフォーマンスデバッグ基盤

### メトリクス収集の必須化
- 全サービスで以下の RED メトリクスを収集する:
  - **Rate**: リクエスト数/秒
  - **Errors**: エラー率（%）
  - **Duration**: レイテンシ（p50, p95, p99）
- DB/キャッシュには USE メトリクスを追加:
  - **Utilization**: CPU/メモリ/ディスク使用率
  - **Saturation**: キュー長、待ち行列数
  - **Errors**: 接続エラー数

### DB パフォーマンス
- クエリ実行計画（EXPLAIN）をログに記録可能にする（本番は閾値超過時のみ）
- インデックス使用状況を定期監視する（未使用インデックスの検出）
- コネクションプールの使用状況をメトリクスとして公開する
  - `pool_size`, `pool_active`, `pool_idle`, `pool_waiting`

### インスタンスパフォーマンス
- CPU / メモリ / ディスクI/O / ネットワークI/O のメトリクスを収集する
- オートスケーリングの閾値とクールダウン期間を文書化する
- スケールイベント（スケールアウト/イン）をログに記録する

### レイテンシの分解
- 全体のレスポンスタイムだけでなく、区間ごとのレイテンシを測定する:
  - ロードバランサー → アプリ（ネットワーク）
  - アプリ内処理（ビジネスロジック）
  - アプリ → DB（クエリ実行）
  - アプリ → 外部API（外部依存）
- ボトルネックの特定に「どの区間が遅いか」の情報が必須

### ベースラインの記録
- 正常時のパフォーマンスベースラインを文書化する
- 異常判定には「通常と比べてどうか」が必要。ベースラインなしでは判断できない

```markdown
## パフォーマンスベースライン（2026-Q1）
| メトリクス | p50 | p95 | p99 |
|-----------|-----|-----|-----|
| API レスポンス | 45ms | 120ms | 350ms |
| DB クエリ | 5ms | 25ms | 80ms |
| 外部API | 200ms | 500ms | 1200ms |
```

## 監査デバッグ基盤

### インフラ変更の監査ログ
- インフラ構成の変更（手動・自動問わず）を全て記録する:
  - インスタンスの起動/停止/スケール
  - セキュリティグループ/ファイアウォール変更
  - DB スキーマ変更（マイグレーション）
  - 環境変数/シークレットの変更
  - デプロイ（誰が・いつ・どのバージョンを）

### DB 監査ログ
- DDL（CREATE/ALTER/DROP）を全て記録する
- 特権操作（GRANT/REVOKE、データの直接UPDATE/DELETE）を記録する
- 大量データ操作（バルクINSERT/DELETE）をアラート対象にする

```sql
-- PostgreSQL: pgaudit 有効化例
-- DDL + 特権操作 + 大量変更を記録
ALTER SYSTEM SET pgaudit.log = 'ddl,role,write';
ALTER SYSTEM SET pgaudit.log_level = 'log';
```

### アクセスログ
- 全サービスへのアクセスログを保持する（最低90日、推奨1年）
- ログに含める情報: タイムスタンプ、送信元IP、メソッド、パス、ステータスコード、レスポンスタイム、ユーザーID
- DB への接続ログ: 接続元IP、ユーザー名、接続/切断タイムスタンプ

### 変更の追跡可能性
- 「誰がいつ何を変えたか」を全レイヤーで追跡可能にする:
  - **アプリケーション**: API監査ログ（`api-sdk.md` 参照）
  - **インフラ**: IaC の Git 履歴 + Cloud Audit Log
  - **DB**: マイグレーション履歴 + pgaudit/Cloud SQL Audit Log
  - **シークレット**: Secret Manager のアクセスログ

## インフラ構成の選定基準

### 選定時のチェックリスト
新しいサービス/ミドルウェア/マネージドサービスを採用する際に確認する:

| # | 確認項目 | 不合格基準 |
|---|---------|-----------|
| 1 | 構造化ログを出力できるか | ログなし or 自由文のみ |
| 2 | メトリクスをエクスポートできるか | CPU/メモリすら取れない |
| 3 | 分散トレースに参加できるか | requestId の伝播が不可能 |
| 4 | 監査ログを有効化できるか | 変更履歴が取れない |
| 5 | アラートを設定できるか | 閾値ベースの通知ができない |
| 6 | ログの保持期間を制御できるか | ログが自動削除される or 保持設定なし |

**上記のいずれかが不合格の場合、代替サービスを検討する。**

### IaC（Infrastructure as Code）必須
- インフラ構成は必ずコード化する（Terraform, Pulumi, CloudFormation等）
- 手動変更は禁止。ドリフト検出を有効にする
- IaC リポジトリにも `docs/README.md` を配置し、構成図・変更手順を記載する

### バックアップとリストア
- DB バックアップの設定を文書化する（頻度、保持期間、リストア手順）
- 四半期ごとにリストアテストを実施する（バックアップは「リストアできて初めて意味がある」）
- リストア手順は `docs/disaster-recovery.md` に記載する
