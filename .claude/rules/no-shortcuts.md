# ワークフローを端折るな

## 絶対原則

**決めたワークフローは端折らない。例外なし。**

「軽微だから」「明らかだから」「時間がないから」は端折る理由にならない。

## 端折ってよい唯一の条件

**同一手順を100回以上実行した実績があり、その手順に意味がないことが実証された場合のみ。**

- 100回未満の経験で「この手順は不要」と判断することは禁止
- 実証とは「手順を省略しても問題が起きなかった」ではなく「手順を実行しても価値が生まれなかった」こと
- 手順の廃止は tech-standards-architect が提案し、CTO が承認する。個人判断で省略しない

## なぜ端折るのか（根本原因）

エージェントがワークフローを端折る理由は以下のいずれか:

| 理由 | 反論 |
|------|------|
| 「軽微な変更だから」 | 軽微かどうかはレビュー・テストを通して初めて分かる。事前に判断できない |
| 「時間がかかるから」 | 手戻りの方が遥かに時間がかかる。4回デプロイし直す方が遅い |
| 「自明だから」 | 自明と思い込んで間違ったファイルを修正した実績がある |
| 「前回うまくいったから」 | 前回と今回は違う。手順は再現性を保証するために存在する |

## 適用範囲

この原則は全エージェント・全ワークフローに適用される:

- 開発ワークフロー（development-workflow.md）の Phase 7-8
- デプロイ前のプレビュー検証
- スクリーンショットの確認手順
- データ追加時のエンティティ確認
- その他 `.claude/rules/` に定められた全手順

## 事故事例（2026-03-27）

CSS修正で development-workflow.md の Phase 7（コードレビュー）・Phase 8（テスト）を端折った結果:
- 間違ったファイルを修正（レビューがあれば1発で防げた）
- モバイルで表示崩れ（テストがあれば1発で防げた）
- 4回デプロイして4回失敗
- ユーザーの信頼を毀損

**端折らずにワークフロー通りやっていれば、1回のデプロイで済んだ。**
