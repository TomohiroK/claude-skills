# プロジェクトドキュメント規約

## docs/ フォルダの必須化
- 開発プロジェクト開始時に `docs/` フォルダを必ず作成する
- ドキュメントなしのコードはメンテナンス不能と見なす

## 必須ドキュメント

### docs/README.md（プロジェクト概要）
- プロジェクトの目的（1-2文）
- 技術スタック一覧
- セットアップ手順（git clone → 動作確認まで）
- ディレクトリ構成（主要フォルダの説明）
- 環境変数一覧（名前・説明・必須/任意。値は書かない）

### カテゴリ単位の README.md
- 主要ディレクトリには `README.md` を配置する
- 最低限の記載内容:
  - このディレクトリの責務（1-2文）
  - 含まれるモジュール/ファイルの一覧と概要
  - 主要な公開関数/コンポーネント/APIの説明
  - 他ディレクトリとの依存関係
- 例:
  - `src/models/README.md` — データモデルの一覧と関係
  - `src/api/README.md` — APIエンドポイントの概要
  - `src/components/README.md` — UIコンポーネントの分類と用途

### docs/architecture.md（規模に応じて）
- システム構成図（テキストベースで可）
- データフロー（入力 → 処理 → 出力）
- 主要な設計判断とその理由（ADR簡易版）

## 記載ルール

### 粒度
- 「ソースコードを読まなくても全体像がわかる」レベル
- 実装詳細は書かない（コードが変わるたびにドキュメントが陳腐化する）
- 「なぜこの構成にしたか」「何に注意すべきか」を重視

### フォーマット
- Markdown 形式
- テーブルを活用して一覧性を高める
- コード例は最小限（使い方を示す程度）

### 更新タイミング
- 新しいディレクトリ/モジュールを追加したとき
- 既存のアーキテクチャに変更があったとき
- README は「作ったら終わり」ではなく、構造変更時に更新する

## 開発作業フローへの組み込み
- 新規プロジェクト作成時: `docs/README.md` + ルートの `README.md` を最初に作成
- 新規ディレクトリ作成時: そのディレクトリに `README.md` を配置
- 機能追加完了時: 関連する README を更新（新モジュールの追記等）
- コードレビュー時: ドキュメントの追加・更新が必要かを確認項目に含める
